## 1. 项目要解决的问题

本项目面向“无人机视角多模态小目标检测”任务，核心困难有三类：

1. 小目标难检测  
   无人机高空俯视导致车辆、行人、三轮车等目标在图像中占比很小，卷积下采样后容易丢失细节。

2. 单模态鲁棒性不足  
   仅使用 RGB 图像时，夜间、低照、雨雾、逆光等条件会显著降低检测精度；红外图可以补充热辐射信息。

3. 场景密集遮挡严重  
   城市道路拥堵时，目标密集且互相遮挡，容易出现漏检和误检。

因此，本项目的总体思路是：

- 用双流网络分别提取 RGB 和 Thermal 特征
- 在中层特征上进行跨模态融合
- 用多尺度特征金字塔增强小目标感知
- 用 Anchor-free 检测头完成目标分类和框回归
- 最终封装成可供前后端调用的推理服务

如果结合最终演示目标，这里的“推理服务”不应只理解为：

- 上传一对图片
- 返回一组框

还应理解为：

- 能接受后端发来的“开始视频/开始流会话”控制命令
- 能从视频文件或实时流中持续取帧
- 能按一定频率做检测
- 能把结果持续回传给后端，再转发给前端页面

## 2. 项目整体架构原理

从原理上看，这个项目可以分为五层：

1. 数据层  
   负责读取对齐的 RGB-T 图像对和标注，并做同步增强。

2. 特征提取层  
   分别从 RGB 和 Thermal 中提取多尺度特征。

3. 融合与增强层  
   对两个模态的特征进行注意力融合，并构建特征金字塔增强小目标语义。

4. 检测层  
   对融合后的多尺度特征进行目标分类、框回归和置信度预测。

5. 服务层  
   将模型推理能力封装为 API，供后端和前端系统调用。

代码目录正好对应这五层：

- `data/`：数据层
- `model/network/`：特征提取、融合和增强层
- `model/detector.py`：检测层总装
- `scripts/`：训练、评估、导出
- `service/`：服务层

如果按最终演示系统去看，还可以再补两层“系统集成层”：

6. 会话控制层  
   由后端负责视频会话、实时流会话的创建、停止、查询和权限控制。

7. 实时展示层  
   由前端负责视频播放、检测框渲染、统计展示和会话切换。

因此整套系统不是“ML 单体服务”，而是：

- ML 负责检测能力本身
- 后端负责业务编排与会话管理
- 前端负责可视化展示与交互

## 3. 数据处理原理与代码对应

### 3.1 为什么 RGB 和 Thermal 必须同步处理

RGB 图像和红外图像是同一场景的两个模态。如果对它们分别做不同的空间变换，例如只对 RGB 做裁剪、不对 Thermal 做裁剪，那么两个模态在空间上就不再对应，融合时会直接出错。

所以原则是：

- 空间变换必须同步
  例如翻转、裁剪、缩放必须同时对 RGB、Thermal 和标注框一起做
- 颜色增强只作用于 RGB
  因为 Thermal 本身不依赖颜色纹理，做 RGB 式色彩扰动没有意义

### 3.2 代码实现位置

#### 数据集类

文件：

- `data/dataset.py`

对应原理：

- `RGBTTargetDataset` 负责读取 RGB 图像、Thermal 图像和标注
- `__getitem__` 中先读入图像对，再读取对应标注
- 返回结构统一为：

```python
{
    "rgb": ...,
    "thermal": ...,
    "targets": ...
}
```

其中 `targets` 包含：

- `boxes`
- `labels`
- `image_id`
- `orig_size`

#### 图像解码和预处理

文件：

- `data/preprocess.py`

对应原理：

- `read_image()`：从磁盘读取图像
- `decode_image_bytes()`：从 API 上传的字节流解码图像
- `ensure_same_size()`：保证 RGB 与 Thermal 尺寸一致
- `resize_pair()`：同步缩放图像与框
- `image_to_tensor()`：转成模型输入张量
- `stack_modalities()`：将 RGB 与 Thermal 组合为 6 通道输入

#### 同步增强

文件：

- `data/transforms.py`

对应原理：

- `RandomHorizontalFlip`：同步水平翻转图像和框
- `RandomVerticalFlip`：同步垂直翻转
- `RandomCrop`：同步裁剪并更新框坐标
- `Resize`：统一缩放到模型输入尺寸
- `ColorJitterRGB`：仅对 RGB 做亮度、对比度、饱和度增强
- `ToTensor`：将增强后的结果转成张量

这里体现的核心思想是：

- 模态配准不能破坏
- 小目标增强尽量保留有效区域
- 数据增强优先提升泛化能力而不是单纯制造随机性

## 4. 双流特征提取原理与代码对应

### 4.1 为什么要做双流 backbone

RGB 和 Thermal 的信息属性不同：

- RGB 擅长表达纹理、颜色、边缘、结构
- Thermal 擅长表达热辐射差异，在夜间和低照场景更稳定

如果一开始就简单把它们拼成一个输入通道直接送入普通 CNN，模型很难显式学习“哪个模态在当前区域更可靠”。

因此，本项目使用双流结构：

- 一条分支专门提取 RGB 特征
- 一条分支专门提取 Thermal 特征
- 在中层再做融合

这种方式更符合多模态学习的基本原则：先分开建模，再进行语义对齐和互补。

### 4.2 代码实现位置

文件：

- `model/network/backbone.py`

对应类：

- `LightweightBranch`
- `RGBTFeatureExtractor`

具体对应关系：

- `LightweightBranch`：单模态轻量 CNN 分支
- `RGBTFeatureExtractor`：内部包含 `rgb_branch` 和 `thermal_branch`

它的执行逻辑是：

1. 输入是 6 通道张量
2. 前 3 通道作为 RGB
3. 后 3 通道作为 Thermal
4. 两个分支分别提取 3 个尺度特征
5. 每个尺度的 RGB/Thermal 特征进入融合模块

这里使用了轻量卷积块 `DepthwiseSeparableBlock`，对应的工程考虑是：

- 降低参数量和计算量
- 保留一定实时性
- 为后续 15+ FPS 目标留出空间

## 5. 跨模态融合原理与代码对应

### 5.1 为什么不能简单相加

RGB 和 Thermal 虽然描述同一场景，但信息分布不同。简单相加或直接拼接的缺点是：

- 无法区分哪个模态在当前位置更可靠
- 夜间时 RGB 特征可能很弱，但简单加和会把噪声也引入
- 白天热干扰或反射场景中，Thermal 也可能存在不稳定区域

因此需要一个“自适应融合权重”机制。

### 5.2 本项目采用的基本思想

本项目使用跨模态注意力融合：

1. 分别对 RGB 和 Thermal 特征做投影
2. 结合两者及它们的差异信息
3. 生成一张注意力图
4. 用这张图决定每个位置更依赖 RGB 还是 Thermal

数学直觉可以理解为：

- 注意力值高：更相信 RGB
- 注意力值低：更相信 Thermal

这比固定比例融合更适合全天候场景。

### 5.3 代码实现位置

文件：

- `model/network/fusion_module.py`

对应类：

- `CrossModalAttentionFusion`

实现逻辑：

```python
rgb = self.rgb_proj(feat_rgb)
thermal = self.thermal_proj(feat_thermal)
fused_context = torch.cat([rgb, thermal, torch.abs(rgb - thermal)], dim=1)
attention = self.attn(fused_context)
fused = attention * rgb + (1.0 - attention) * thermal
```

这里的关键点：

- `torch.abs(rgb - thermal)`  
  用于显式建模两个模态的差异区域

- `attention * rgb + (1 - attention) * thermal`  
  表示可学习的动态模态加权

这部分就是你方案里“融合模块创新点”的核心代码落点。

## 6. 小目标增强原理与代码对应

### 6.1 为什么小目标需要多尺度增强

小目标本身像素少，经过多次下采样后容易出现：

- 特征模糊
- 目标和背景难以区分
- 框回归不稳定

因此，检测小目标通常需要：

- 保留更高分辨率特征
- 在不同尺度上传递语义信息
- 对浅层细节和深层语义做融合

### 6.2 本项目中的做法

本项目采用两层增强思路：

1. 轻量 BiFPN 风格的特征金字塔  
   用于在多尺度之间传递特征

2. 小目标特征精炼头  
   通过不同膨胀率卷积进一步扩大感受野，同时保留局部细节

### 6.3 代码实现位置

#### 多尺度特征金字塔

文件：

- `model/network/neck.py`

对应类：

- `LightweightBiFPN`

代码逻辑：

- 将 backbone 输出的不同尺度特征统一到相同通道数
- 自顶向下融合高层语义
- 自底向上再融合底层细节

这是典型的小目标增强思路，因为：

- 高层特征语义强
- 低层特征定位细
- 二者结合有助于提升小目标检测稳定性

#### 小目标精炼头

文件：

- `model/network/head.py`

对应类：

- `SmallObjectRefineHead`

实现逻辑：

- 对 3 个尺度分别做卷积增强
- 不同层用不同膨胀率

这样做的含义是：

- 低层更关注局部边缘细节
- 高层更关注上下文感受野
- 对密集场景和小目标都有帮助

## 7. 检测头原理与代码对应

### 7.1 为什么选 Anchor-free

传统 Anchor-based 检测器需要人工设置大量先验框，而无人机交通场景中：

- 目标尺度变化大
- 小目标很多
- 密集遮挡多

Anchor 配置不合适时很容易造成正负样本分配不稳定。

Anchor-free 方法的优势是：

- 不依赖复杂 anchor 设计
- 正样本分配更简洁
- 对小目标和密集目标更容易快速迭代

因此当前参考实现采用 `FCOS`。

### 7.2 代码实现位置

文件：

- `model/detector.py`

对应类：

- `RGBTBackboneWithNeck`
- `RGBTDetector`

原理和代码关系如下：

#### `RGBTBackboneWithNeck`

作用：

- 先调用双流特征提取
- 再调用融合模块
- 然后进入 BiFPN
- 最后通过小目标精炼头输出多尺度特征

#### `RGBTDetector`

作用：

- 将上面的多尺度特征交给 `torchvision.models.detection.FCOS`
- 完成分类、框回归和 centerness 预测

这里其实是“自定义 backbone + 标准检测器头部”的工程实现方式，优点是：

- 能快速搭好训练和推理闭环
- 保留结构创新空间
- 避免从零重写整套检测器训练逻辑

### 7.3 模型输入为什么是 6 通道

在 `RGBTDetector.forward()` 中：

```python
images = [stack_modalities(rgb, thermal) for rgb, thermal in zip(rgb_images, thermal_images)]
```

含义是：

- 每个样本由 `RGB(3)` 和 `Thermal(3)` 拼成 `6` 通道输入
- 进入自定义 backbone 后再拆开建模

这么做的目的只是为了兼容 `torchvision detection` 的统一输入格式，真正的双流处理是在 `RGBTFeatureExtractor` 内部完成的。

## 8. 损失函数原理与代码对应

### 8.1 为什么要关注小目标损失

比赛要求强调小目标性能，因此训练时不能只看整体 loss，还要让模型更关注小目标样本。

普通检测训练中，大目标更容易拟合，小目标往往贡献较弱。为了避免模型“偏向容易样本”，这里增加了一个小目标比例感知的损失聚合策略。

### 8.2 代码实现位置

文件：

- `model/loss.py`

对应内容：

- `count_small_objects()`
- `SmallObjectLossAggregator`

核心思想：

1. 统计一个 batch 中小目标占比
2. 如果小目标占比较高，则适度提升分类损失权重
3. 让模型在训练时更关注小目标识别

这里不是完全重写 FCOS 内部 loss，而是在外层对 loss 做聚合和再加权，这是一个比较务实的工程方案：

- 兼容现有检测器
- 修改成本低
- 容易根据实验继续调整

## 9. 训练流程原理与代码对应

### 9.1 训练流程基本逻辑

一次完整训练包含：

1. 读取配置
2. 构建训练集和验证集
3. 构建模型
4. 前向传播得到损失
5. 反向传播更新参数
6. 周期性验证
7. 保存 `last.pt` 和 `best.pt`

### 9.2 代码实现位置

文件：

- `scripts/train.py`

核心函数说明：

- `load_config()`：读取 YAML 配置
- `build_dataloaders()`：构建训练/验证数据加载器
- `move_targets_to_device()`：将标注移动到 GPU/CPU
- `validate()`：在验证集上做推理并计算指标
- `main()`：总训练入口

### 9.3 为什么要保存 best 和 last

- `last.pt`：保存最近一次训练状态，便于断点续训
- `best.pt`：保存验证性能最优权重，便于最终部署和答辩展示

这在比赛工程中是标准做法。

## 10. 评估指标原理与代码对应

### 10.1 为什么不能只看总体 mAP

赛题特别强调：

- 多模态融合是否优于单模态
- 小目标性能是否提升

所以评估不能只给一个总体数值，还要关注：

- `mAP@0.5`
- `Recall@0.5`
- `small_recall@0.5`
- 分类别 AP

### 10.2 代码实现位置

文件：

- `scripts/eval_utils.py`
- `scripts/evaluate.py`

对应原理：

- `box_iou()`：计算预测框与真值框 IoU
- `_match_single_image()`：根据 IoU 和类别做匹配
- `_ap_from_ranked()`：根据排序结果计算 AP
- `evaluate_predictions()`：聚合整套评估结果

这里的 `small_recall50` 是专门针对小目标做的补充指标，用来对应赛题中的小目标优化要求。

## 12. API 服务原理与代码对应

### 12.1 为什么需要服务化

模型代码本身只能在本地脚本里运行，但项目最终需要：

- 后端系统通过 HTTP 调用算法
- 前端展示检测结果
- 支持统一部署和版本管理

所以必须把模型封装成 API 服务。

### 12.2 代码实现位置

文件：

- `service/api/app.py`
- `service/core/schemas.py`

对应接口：

#### `GET /v1/health`

作用：

- 供后端健康检查
- 判断模型是否加载成功

#### `GET /v1/model/info`

作用：

- 返回模型名称、版本、类别映射、输入尺寸
- 便于后端记录和前端展示

#### `POST /v1/detect/stream`

作用：

- 接收一对 RGB/Thermal 图像
- 返回检测结果

请求方式：

- `multipart/form-data`

请求字段：

- `rgb_image`
- `thermal_image`
- `request_id`

响应模型定义在：

- `DetectionItem`
- `DetectionResponse`

这样做的好处是：

- 接口字段固定
- 便于后端反序列化
- 便于前端直接绘制检测框

### 12.3 当前实现和最终演示目标的关系

这一点必须说清楚。

当前仓库代码里，真正已经实现并可直接运行的是：

- `GET /v1/health`
- `GET /v1/model/info`
- `POST /v1/detect/stream`

其中 `POST /v1/detect/stream` 的本质是：

- 接收一对 RGB/Thermal 图像文件
- 做一次检测
- 返回一次 JSON 结果

所以它更适合：

- 模型验收
- 单张样例调试
- 前后端接口冒烟测试

它还不是最终演示所需的“实时视频会话推理接口”。

最终演示目标下，系统还需要具备：

- 视频文件会话启动/停止
- 实时流会话启动/停止
- 会话状态管理
- 持续拉流与逐帧推理
- 结果回调到后端
- 后端再通过 WebSocket 推送到前端

### 12.4 面向最终演示的实时视频推理封装原理

如果把最终演示链路按职责拆开，可以分成下面七步：

1. 设备或视频源进入平台  
   设备双路 RGB/IR 推流到流媒体服务，或者用户上传视频文件。

2. 后端创建推理会话  
   后端根据前端操作创建 `video session` 或 `stream session`，生成 `sessionId`。

3. 后端通知模型端开始工作  
   后端把 `sessionId`、`sourceUrl`、`streamKey`、回调地址等信息发给 ML 服务。

4. 模型端持续读取视频/流  
   ML 服务根据 `sourceUrl` 拉取视频或实时流，持续解码、抽帧、组织成待检测帧。

5. 模型端执行逐帧检测  
   每拿到一帧或一对 RGB-T 帧，就走一次和当前 `Predictor.predict()` 相同的检测主流程。

6. 模型端持续回调检测结果  
   ML 服务把每次检测结果连同 `sessionId`、时间戳、帧序号等信息回调给后端。

7. 后端转发给前端展示  
   后端完成落库、统计、权限校验后，再通过 WebSocket 或订阅通道推送给前端。

这里面最关键的思想是：

- 视频播放链路和检测结果链路不是同一件事

更准确地说：

- 一条链路负责“把视频播出来”
- 一条链路负责“把检测结果叠加上去”

对前端而言，常见做法是：

- 播放器自己播放视频或实时流
- 检测框通过另外一条实时消息链路叠加到播放器画面上

### 12.5 在最终演示里，ML 侧应该负责什么

如果按你当前的角色边界，ML 侧需要负责的不是整套业务系统，而是下面这些模块：

1. 模型加载与推理主流程  
   也就是当前 `Predictor + InferenceEngine` 这一层。

2. 视频/流读取与抽帧  
   包括：
   - 从 `sourceUrl` 拉流或读取视频文件
   - 控制抽帧频率
   - 把帧转成模型可接收的输入

3. RGB-T 双路帧对齐  
   如果现场是双路 RGB/IR 流，ML 或其前置模块必须保证送进模型的两路帧是时间上基本对齐的。

4. 会话生命周期响应  
   能响应：
   - `startVideoSession`
   - `startStreamSession`
   - `stopSession`

5. 检测结果回调  
   能把 `sessionId + detections + inference_time + 时间戳/帧号` 稳定回调给后端。

6. 推理频率和资源控制  
   包括：
   - 是否每帧都推理
   - 是否按固定 FPS 采样
   - 队列满时丢帧还是阻塞

### 12.6 在最终演示里，ML 侧不应该负责什么

为了和前后端同学边界清晰，下面这些通常不属于 ML 核心职责：

- 用户登录与鉴权
- 设备管理和设备绑定
- `app + streamKey` 路由维护
- WebSocket ticket 发放
- 前端页面播放控件
- 页面上的告警、报表、统计筛选

也就是说，ML 侧不要把自己做成一个“大而全的业务后端”。

更合理的定位是：

- 一个能被后端编排的推理执行器

### 12.7 小项目下的推荐最小可交付方案

考虑到你们是小项目、你只负责 ML，建议不要一口气做成全功能流媒体平台，而是优先做一个最小可交付闭环：

1. 保留当前单次图像对接口  
   用于模型冒烟测试和单帧验收。

2. 在此基础上补“会话式推理执行器”  
   只需要支持：
   - `startVideoSession`
   - `startStreamSession`
   - `stopSession`

3. 模型端内部按固定频率抽帧  
   例如每秒 2 到 5 帧，不追求逐帧满帧推理。

4. 结果异步回调后端  
   不要求模型端自己管理前端长连接。

5. 前端用播放器负责显示视频  
   再叠加后端推来的检测框。

这样设计的好处是：

- 业务边界清楚
- 工程复杂度可控
- 足够支撑联调、验收和阶段性演示
- 也更符合你“只负责 ML 部分”的现实分工

## 13. 导出与部署原理与代码对应

### 13.1 为什么要导出模型

训练用的 `.pt` 权重便于继续训练，但部署阶段常常还需要：

- TorchScript：便于 PyTorch 推理部署
- ONNX：便于跨平台或接 TensorRT / ONNX Runtime

### 13.2 代码实现位置

文件：

- `scripts/export.py`
- `service/utils/inference_engine.py`

对应原理：

- `SingleBatchExportWrapper`：将检测输出包装为导出友好的形式
- `export.py`：导出 TorchScript 和 ONNX
- `TorchInferenceEngine`：当前默认 PyTorch 推理引擎
- `ONNXRuntimeEngine`：为后续 ONNX Runtime 加速预留入口

这部分对应工程部署阶段的“模型导出与加速”能力。

## 14. 配置管理原理与代码对应

### 14.1 为什么要用配置文件

训练和部署中涉及大量参数：

- 数据目录
- 类别数
- 输入尺寸
- 学习率

其中：

- `default.yaml`：训练、模型、数据、导出配置

这种方式的价值在于：

- 实验可复现
- 调参更方便
- 团队协作时修改边界更清晰

## 15. 与前后端衔接的基本原理

### 15.1 给后端的核心数据

后端最关心的是“调用一次算法能拿到什么结果”，因此 ML 侧需要提供稳定结构：

```json
{
  "request_id": "abc123",
  "detections": [
    {
      "bbox": [120.5, 210.2, 168.9, 250.1],
      "confidence": 0.91,
      "class_id": 0,
      "class_name": "car"
    }
  ],
  "inference_time": 0.032,
  "model_input_size": [640, 640],
  "image_size": [1920, 1080],
  "model_version": "best"
}
```

后端可基于它实现：

- 请求日志跟踪
- 检测结果落库
- 统计分析
- 任务状态管理

### 15.2 给前端的核心数据

前端最关心的是“怎么画框和展示信息”，因此 ML 返回字段要明确：

- `bbox`：原图坐标
- `class_name`：显示标签
- `confidence`：显示置信度
- `image_size`：用于缩放映射
- `model_version`：用于版本说明

这样前端就能直接实现：

- 图片框选展示
- 类别颜色区分
- 检测数量统计
- 置信度排序列表

### 15.3 实时视频演示时，给后端的新增核心数据

如果是实时视频/实时流推理，后端只拿到单次 `DetectionResponse` 是不够的。

后端还需要至少以下附加上下文：

- `session_id`
- `source_type`
  - `video` 或 `stream`
- `frame_index`
- `frame_timestamp`
- `stream_key` 或视频对象标识
- `is_end`
  - 用于表示视频会话是否完成

一个更贴近最终演示的回调体可以抽象成：

```json
{
  "session_id": "sid_xxx",
  "source_type": "stream",
  "frame_index": 128,
  "frame_timestamp": 1711800000.125,
  "detections": [
    {
      "bbox": [120.5, 210.2, 168.9, 250.1],
      "confidence": 0.91,
      "class_id": 0,
      "class_name": "car"
    }
  ],
  "inference_time": 0.032,
  "image_size": [1920, 1080],
  "model_version": "best"
}
```

这样后端才能进一步做：

- 会话级结果聚合
- 时序展示
- WebSocket 推送
- 历史记录查询

### 15.4 实时视频演示时，给前端的正确解释口径

前端同学最容易混淆的一点是：

- “视频怎么播放”
- “检测结果怎么叠加”

实际上这是两条链路。

#### 视频播放链路

前端播放器自己负责：

- 拉取播放地址
- 解码播放视频
- 管理暂停、播放、切换

#### 检测结果链路

前端另外再从后端拿：

- 当前会话的检测框
- 当前帧对应的类别与置信度
- 实时统计数据

然后再把检测框叠加到播放器画面上。

这就是为什么最终联调时，前端不应该直接把播放器流地址当成“算法输入接口”。

### 15.5 你作为 ML 负责人，对前后端最需要讲清楚的三件事

1. 当前仓库现状  
   当前仓库已经实现：
   - 单次 RGB-T 图像对检测
   - 视频/实时流会话启动与停止
   - HTTP-FLV / 视频文件读取
   - 抽帧推理
   - 结构化结果回调

   但默认仍然是“结果回调 + 前端叠加”的链路，不是“模型端直接回传带框视频流”。

2. 最终联调目标  
   最终需要的是“会话控制 + 拉流/读视频 + 抽帧推理 + 结果回调 + 前端叠加展示”的闭环；后续如需升级，再补“模型端在线标注视频流”。

3. 你负责的边界  
   你负责：
   - 模型
   - 推理执行
   - 输入规整
   - 结果输出

   前后端负责：
   - 会话业务
   - 用户权限
   - 播放器
   - 页面展示
   - 数据落库

## 17. 后续可继续强化的方向

如果后续目标是进一步提升效果和工程完成度，可以继续从以下方向增强：

1. 更强的融合结构

6. 更完整的视频链路  
   补“模型端在线标注视频流”、归一化坐标字段、前端播放器直接消费带框流的工程链路。
