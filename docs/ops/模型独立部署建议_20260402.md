# 模型独立部署建议（2026-04-02）

## 1. 问题背景

当前项目要满足比赛题目里的核心目标：

- 前后端联动
- 实时视频演示
- 模型端主动拉流并做检测

Tmiya 当前的条件是：

- 已有一台阿里云轻量应用服务器
- 前后端准备部署到这台服务器
- 想知道模型服务是否也应该部署到这台服务器

## 2. 先给结论

最推荐的方案是：

`前后端 + SRS / 流媒体服务部署到阿里云轻量服务器，模型服务独立部署，不要现在就和前后端一起塞到轻量服务器里。`

如果你自己的电脑可以开放 GPU，那么当前阶段最实用的做法是：

`把模型服务部署在你自己的电脑上，用你本机 GPU 跑推理；前后端和流媒体放云上。`

## 3. 为什么不建议把模型也塞进轻量服务器

不是说轻量服务器完全不能跑，而是说：

- 它更适合承载前端页面、后端接口、SRS 这类业务和流媒体模块
- 不适合当前就承担实时视频推理主负载

原因主要有三点：

### 3.1 当前 CPU 侧推理速度还不够理想

你已经做过 `HTTP-FLV` 的 `5min` 稳定性测试，结果是：

- 测试时长约 `301.242s`
- 处理帧数 `223`
- 等效处理速度约 `0.7403 fps`
- 最大连续停滞约 `11s`

这说明：

- 链路是可用的
- 但 CPU 跑实时视频检测仍然偏慢

### 3.2 阿里云轻量服务器大概率没有 GPU

如果把模型推理放在没有 GPU 的轻量服务器上，通常会面临：

- 推理速度慢
- 和 SRS/后端抢 CPU
- 内存压力更大
- 视频展示和检测回调更容易互相影响

### 3.3 一台机器上堆太多模块，排障会非常难

如果把这些都压在一台轻量服务器上：

- 前端
- 后端
- SRS / 流媒体
- 模型服务

那一旦出现：

- 播放卡顿
- 检测掉帧
- 回调超时
- CPU 爆满

你会很难快速判断到底是哪里出了问题。

## 4. 你的电脑开放 GPU 后，为什么更适合做模型端

你现在的 `0.74 fps` 是基于：

- `CPU`
- `HTTP-FLV` 拉流
- 模型推理
- Python 调度

这一整条链路测出来的。

它不代表你开放 GPU 后的最终速度。

如果模型推理放到你本机 GPU 上，通常会明显改善：

- 模型前向速度
- 后处理速度
- 单帧整体推理耗时

虽然：

- 拉流
- 解码
- 网络波动

这些问题不会因为 GPU 完全消失，但至少“推理慢”这一块会缓解很多。

所以，从比赛演示角度：

- 前后端上云
- 模型用本机 GPU

通常是更现实的组合。

## 5. 推荐部署架构

建议采用下面这个结构：

### 云上轻量服务器

负责：

- 前端页面
- 后端业务服务
- SRS / 流媒体服务
- 登录注册、会话管理、结果转发

### 你自己的电脑

负责：

- 模型推理服务
- 使用本机 GPU
- 主动拉取云上 `HTTP-FLV`
- 拆成 `RGB + Thermal` 图片对
- 检测
- 回调后端

这条链路可以概括成：

1. 视频源推到云上流媒体
2. 前端从云上看视频
3. 后端创建推理会话
4. 后端调用你本机上的模型服务
5. 模型服务主动拉云上的 `HTTP-FLV`
6. 检测结果回调给云上后端
7. 后端再推给前端叠框

## 6. 如果采用模型独立部署，你需要准备什么

如果模型服务部署在你自己的电脑上，至少要准备这些：

### 6.1 模型服务启动

在本机启动模型服务，例如：

```bash
cd /home/liujuncheng/rgbt_uav_detection
uvicorn service.api.app:app --host 0.0.0.0 --port 8000
```

### 6.2 本机可被云上后端访问

后端必须能访问到你的模型服务地址。

常见方式：

- 你本机有公网 IP
- 或者你做内网穿透

可选方案：

- `frp`
- `ngrok`
- `cloudflared tunnel`
- `Tailscale`

### 6.3 开放模型服务端口

至少要放通：

- 模型服务 HTTP 端口，例如 `8000`

### 6.4 后端需要知道你的模型服务地址

例如：

- `http://你的地址:8000/v1/inference/stream/start`

## 7. 后端还需要给模型端什么

如果做正式联调，后端至少要给你这些：

- `source_url`
- `resultCallbackUrl`
- `X-Model-Token` 实际值
- `session_id`
- `app`
- `stream_key`
- `pair_layout`
- `rgb_position`

其中最关键的两个是：

1. 模型端主动拉流地址
2. 模型端回调后端地址

## 8. 是否可以全部部署到一台轻量服务器

可以，但只建议作为“最低配演示方案”。

这意味着你要接受：

- 推理帧率低
- 框更新慢
- 可能有明显抖动
- 多模块互相抢资源

这种方案适合：

- 先把所有模块都摆出来
- 勉强演示一条闭环

不适合：

- 追求较好的实时感
- 长时间稳定演示

## 9. 比赛题目要求下的综合建议

从比赛要求看，老师或评委更在意的是：

- 你有没有完整系统链路
- 视频能不能实时展示
- 检测结果能不能稳定叠加
- 前后端和模型是否联动

他们通常不要求你必须把所有模块都塞到一台服务器。

所以更合理的答题方式是：

- 云上轻量服务器承担平台和流媒体职责
- 模型服务单独部署到更适合推理的环境
- 保证整条系统链路稳定跑通

这比为了“都放一台机”而牺牲实时效果更符合比赛目标。

## 10. 当前最稳妥的执行方案

建议直接按下面顺序推进：

1. 前端 + 后端 + SRS 先部署到阿里云轻量服务器
2. 模型服务先部署到你本机 GPU
3. 后端接入模型端 `start/stop/query` 接口
4. 模型端主动拉取云上的 `HTTP-FLV`
5. 模型端回调结果给后端
6. 前端页面叠框展示
7. 等训练和推理优化后，再决定是否迁移模型服务

## 11. 一句话结论

`前后端部署到一台阿里云轻量服务器可以，但模型服务当前最适合独立部署，优先放在你自己的 GPU 机器上。`
