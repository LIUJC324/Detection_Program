## 4. 明天如果要继续训练，推荐怎么做

### 4.1 直接继续当前权重

仓库里已有基础续训脚本：

```bash
cd /home/liujuncheng/rgbt_uav_detection
bash scripts/resume_training.sh
```

它默认使用：

- `configs/default.yaml`
- `weights/last.pt`

### 4.2 更推荐的做法：重新开新的 tmux 会话和新日志

为避免再次覆盖背景日志，建议用下面这个脚本：

```bash
cd /home/liujuncheng/rgbt_uav_detection
bash scripts/restart_refined_training_tmux.sh
```

这个脚本会：

1. 新开一个 `tmux` 会话
2. 默认从 `weights/last.pt` 恢复
3. 自动生成新的时间戳日志文件
4. 不再复用 `outputs/train_tmux_background.log`

默认会话名：

- `rgbt_train_resume`

默认日志示例：

- `outputs/train_resume_YYYYMMDD_HHMMSS.log`

如果需要自定义：

```bash
SESSION_NAME=rgbt_train_stage2 \
CHECKPOINT_PATH=/home/liujuncheng/rgbt_uav_detection/weights/last.pt \
LOG_PATH=/home/liujuncheng/rgbt_uav_detection/outputs/train_resume_stage2.log \
bash scripts/restart_refined_training_tmux.sh
```

## 6. 明天接手时最容易踩的坑

### 坑 1：误以为日志“回退了”

原因通常是：

- 后台日志文件被覆盖重建
- 新 run 从另一个 checkpoint 起跑

规避方式：

- 优先看 `outputs/train_runs/run_*/`
- 不要只看单一背景日志文件
- 明天优先使用新的 `restart_refined_training_tmux.sh`

### 坑 2：旧指标和新数据口径混在一起

当前已经处理过：

- 如果 checkpoint 的数据配置和当前配置不同，`train.py` 会重置旧的 `resume history`

但仍建议：

- 每一轮新实验都保留新的 `run_YYYYMMDD_HHMMSS`
- 不要手工覆盖 run 目录

### 坑 3：以为训练结束后还会自动继续

当前没有保留任何“自动继续训练”的定时器。

因此：

- 这轮到 `epoch 22` 就停
- 明天需要人工决定是否继续训练
