## 2. 已确认根因

### 2.1 直接根因：送入 FCOS 的特征尺度不对

修复前，`640 x 640` 输入下，主干输出三层特征尺寸为：

- `320 x 320`
- `160 x 160`
- `80 x 80`

这对应的实际步长大约是：

- `2`
- `4`
- `8`

但 `torchvision FCOS` 的匹配、中心采样和 box coding 设计，默认假设的是更接近：

- `8`
- `16`
- `32`

这类检测金字塔。

问题在于：

1. 当前特征图过细，候选点密度过高
2. 回归头更容易退化成“很多高分点响应 + 几乎没有有效框尺寸”
3. 与 FCOS 的正样本尺度分配逻辑不匹配

这也是为什么会出现：

- 预测数量很多
- 分数不低
- 但框退化成点
- `recall50` 一直是 `0`

### 2.2 已排除的问题

下面这些不是这次的主根因：

- 数据类别从 `0` 开始这一点本身没错
  - 当前 `torchvision FCOS` 这里是按 `0..C-1` 的 label 做 one-hot
- 单纯把 score threshold 降低不能解决问题
  - 从 `0.05` 扫到 `0.5`，`TP` 仍然是 `0`
- 前后端接口也不是原因
  - 当前问题发生在训练 / 评估主链路内部

## 6. 开源项目学习与借鉴顺序

下面这些仓库不是只留作参考链接，而是后续应该直接拆模块学习。

### 6.1 第一优先级：UA-CMDet

仓库：

- https://github.com/SunYM2020/UA-CMDet

为什么第一优先级最高：

1. 任务和 `DroneVehicle` 最接近
2. 同样是 RGB-Infrared / UAV 车辆检测路线
3. 更适合借鉴训练口径、样本质量建模和后处理

最值得学的点：

- uncertainty-aware training
- illumination-aware cross-modal NMS
- DroneVehicle 场景下的数据与训练组织方式

当前项目里最适合借鉴的位置：

- `scripts/train.py`
- 后处理与结果融合逻辑

### 6.2 第二优先级：CALNet-Dronevehicle

仓库：

- https://github.com/hexiao0275/CALNet-Dronevehicle

为什么紧跟其后：

1. 它直接针对跨模态语义冲突
2. 对当前项目的融合模块最有启发

最值得学的点：

- `CCR`：Cross-Modal Conflict Rectification
- `SCF`：Selected Cross-modal Fusion

当前项目里最适合借鉴的位置：

- `model/network/fusion_module.py`

建议做法：

- 下一轮优先把当前轻量 attention fusion 升级成“先纠冲突，再融合”的结构

### 6.3 第三优先级：CFT / multispectral-object-detection

仓库：

- https://github.com/DocF/multispectral-object-detection

为什么值得看：

1. 它更强调全局上下文与跨模态长距离交互
2. 适合在当前框架稳定之后，再考虑增强中高层融合

最值得学的点：

- Transformer 风格跨模态交互
- 更强的中高层上下文建模

当前项目里最适合借鉴的位置：

- `model/network/fusion_module.py`
- `model/network/neck.py`

### 6.4 第四优先级：EAEFNet

论文与代码线索：

- https://freeformrobotics.org/wp-content/uploads/2023/04/RAL-22-3446.pdf
- 论文中给出的代码地址：`https://github.com/FreeformRobotics/EAEFNet`

为什么值得看：

1. 它强调 RGB 和 Thermal 并不总是同时可靠
2. 适合当前项目这类白天 / 夜间 / 弱模态交替主导场景

最值得学的点：

- 弱模态补偿
- 双分支显式融合逻辑

当前项目里最适合借鉴的位置：

- `model/network/fusion_module.py`

### 6.5 学习顺序建议

不建议同时学太多路线，建议按下面顺序拆：

1. 先看 `UA-CMDet` 的训练与后处理
2. 再看 `CALNet` 的融合模块
3. 然后看 `EAEF` 的弱模态补偿
4. 最后再考虑 `CFT` 这类更重的全局交互

原因：

- 当前项目先要把“能学出框”做出来
- 不是先追最重的结构

## 12. 空盒问题的模型端优化

### 12.1 现象区分

联调里要区分两个问题：

1. `boxes: []` 且 `error: ""`
2. `boxes` 非空，但有大量“空盒 / 线盒 / 零面积框”

这两种不是一回事。

### 12.2 `boxes: []` 的真实含义

这表示：

- 这一帧推理与回调流程是成功的
- 但最终没有可回调的框

不是：

- 接口报错
- 回调失败

### 12.3 本次已做的空盒优化

已改文件：

- `service/core/predictor.py`
- `service/streaming/session_manager.py`
- `configs/deploy_integration_stable.yaml`

已做的事：

1. 保留 `result["detections"]` 作为主输出
2. 额外保留 `raw_detections` 供回调阶段做低风险 fallback
3. 回调前先过滤退化框：
   - `x2 - x1 <= callback_drop_degenerate_size`
   - 或 `y2 - y1 <= callback_drop_degenerate_size`
4. 如果主阈值筛完后为空，再从 `raw_detections` 里用更低阈值补一次

当前联调配置：

- `callback_min_confidence: 0.65`
- `callback_fallback_min_confidence: 0.45`
- `callback_drop_degenerate_size: 2`

这一步的目标不是“提高学术指标”，而是：

- 降低 `boxes: []` 的出现频率
- 降低前端看到的退化空盒
- 同时不改接口字段

### 12.4 当前仍然存在的限制

即使加了 fallback，当前模型仍然有明显局限：

1. 有些帧仍然会出现 `boxes: []`
2. 原因不是回调坏了，而是该帧确实没有通过阈值和退化框过滤
3. 当前主问题依然是误检过多，而不是只靠调阈值就能彻底解决

## 14. 性能与抽帧推理建议

### 14.1 前后端同学提到的“每隔几帧推一次”是什么意思

这件事当前其实已经有现成入口，不需要改接口。

当前流 / 视频启动接口里本来就有：

- `sampleFps`

它的意义是：

- 控制每秒抽多少帧做推理

所以“每隔几帧推一次”在当前系统里，等价于：

- 把 `sampleFps` 调低

### 14.2 为什么前端会感觉“100ms 一下子出来好几条”

因为：

1. `modelLatencyMs` 表示单帧推理时间
2. 它不等于相邻回调之间的固定时间间隔
3. 流缓冲、解码和异步回调会让结果成簇到达

因此看到：

- 一帧大概 `100ms`
- 但短时间连续收到多条结果

这是流式异步链路的正常现象。

### 14.3 当前联调建议

如果前端觉得：

- 结果太密
- 页面太乱
- GPU 压力偏大

建议优先改：

- `sampleFps`

推荐口径：

1. 稳定联调优先：
   - `sampleFps = 1`
2. 展示流畅与压力折中：
   - `sampleFps = 2`
3. 细粒度观察优先：
   - `sampleFps = 3`

不建议在当前阶段继续无节制提高：

- 推理频率

因为当前主要瓶颈不是“推理不够频繁”，而是“结果质量还不够稳定”。
