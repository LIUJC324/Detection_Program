## 2. 已确认根因

### 2.1 直接根因：送入 FCOS 的特征尺度不对

修复前，`640 x 640` 输入下，主干输出三层特征尺寸为：

- `320 x 320`
- `160 x 160`
- `80 x 80`

这对应的实际步长大约是：

- `2`
- `4`
- `8`

但 `torchvision FCOS` 的匹配、中心采样和 box coding 设计，默认假设的是更接近：

- `8`
- `16`
- `32`

这类检测金字塔。

问题在于：

1. 当前特征图过细，候选点密度过高
2. 回归头更容易退化成“很多高分点响应 + 几乎没有有效框尺寸”
3. 与 FCOS 的正样本尺度分配逻辑不匹配

这也是为什么会出现：

- 预测数量很多
- 分数不低
- 但框退化成点
- `recall50` 一直是 `0`

### 2.2 已排除的问题

下面这些不是这次的主根因：

- 数据类别从 `0` 开始这一点本身没错
  - 当前 `torchvision FCOS` 这里是按 `0..C-1` 的 label 做 one-hot
- 单纯把 score threshold 降低不能解决问题
  - 从 `0.05` 扫到 `0.5`，`TP` 仍然是 `0`
- 前后端接口也不是原因
  - 当前问题发生在训练 / 评估主链路内部

## 6. 开源项目学习与借鉴顺序

下面这些仓库不是只留作参考链接，而是后续应该直接拆模块学习。

### 6.1 第一优先级：UA-CMDet

仓库：

- https://github.com/SunYM2020/UA-CMDet

为什么第一优先级最高：

1. 任务和 `DroneVehicle` 最接近
2. 同样是 RGB-Infrared / UAV 车辆检测路线
3. 更适合借鉴训练口径、样本质量建模和后处理

最值得学的点：

- uncertainty-aware training
- illumination-aware cross-modal NMS
- DroneVehicle 场景下的数据与训练组织方式

当前项目里最适合借鉴的位置：

- `scripts/train.py`
- 后处理与结果融合逻辑

### 6.2 第二优先级：CALNet-Dronevehicle

仓库：

- https://github.com/hexiao0275/CALNet-Dronevehicle

为什么紧跟其后：

1. 它直接针对跨模态语义冲突
2. 对当前项目的融合模块最有启发

最值得学的点：

- `CCR`：Cross-Modal Conflict Rectification
- `SCF`：Selected Cross-modal Fusion

当前项目里最适合借鉴的位置：

- `model/network/fusion_module.py`

建议做法：

- 下一轮优先把当前轻量 attention fusion 升级成“先纠冲突，再融合”的结构

### 6.3 第三优先级：CFT / multispectral-object-detection

仓库：

- https://github.com/DocF/multispectral-object-detection

为什么值得看：

1. 它更强调全局上下文与跨模态长距离交互
2. 适合在当前框架稳定之后，再考虑增强中高层融合

最值得学的点：

- Transformer 风格跨模态交互
- 更强的中高层上下文建模

当前项目里最适合借鉴的位置：

- `model/network/fusion_module.py`
- `model/network/neck.py`

### 6.4 第四优先级：EAEFNet

论文与代码线索：

- https://freeformrobotics.org/wp-content/uploads/2023/04/RAL-22-3446.pdf
- 论文中给出的代码地址：`https://github.com/FreeformRobotics/EAEFNet`

为什么值得看：

1. 它强调 RGB 和 Thermal 并不总是同时可靠
2. 适合当前项目这类白天 / 夜间 / 弱模态交替主导场景

最值得学的点：

- 弱模态补偿
- 双分支显式融合逻辑

当前项目里最适合借鉴的位置：

- `model/network/fusion_module.py`

### 6.5 学习顺序建议

不建议同时学太多路线，建议按下面顺序拆：

1. 先看 `UA-CMDet` 的训练与后处理
2. 再看 `CALNet` 的融合模块
3. 然后看 `EAEF` 的弱模态补偿
4. 最后再考虑 `CFT` 这类更重的全局交互

原因：

- 当前项目先要把“能学出框”做出来
- 不是先追最重的结构

## 12. 空盒问题的模型端优化

### 12.1 现象区分

联调里要区分两个问题：

1. `boxes: []` 且 `error: ""`
2. `boxes` 非空，但有大量“空盒 / 线盒 / 零面积框”

这两种不是一回事。

### 12.2 `boxes: []` 的真实含义

这表示：

- 这一帧推理与回调流程是成功的
- 但最终没有可回调的框

不是：

- 接口报错
- 回调失败

### 12.3 本次已做的空盒优化

已改文件：

- `service/core/predictor.py`
- `service/streaming/session_manager.py`
- `configs/deploy_integration_stable.yaml`

   - 或 `y2 - y1 <= callback_drop_degenerate_size`

- 降低 `boxes: []` 的出现频率

即使加了 fallback，当前模型仍然有明显局限：

1. 有些帧仍然会出现 `boxes: []`
2. 原因不是回调坏了，而是该帧确实没有通过阈值和退化框过滤

## 14. 性能与抽帧推理建议

### 14.1 前后端同学提到的“每隔几帧推一次”是什么意思

这件事当前其实已经有现成入口，不需要改接口。

当前流 / 视频启动接口里本来就有：

- `sampleFps`

它的意义是：

- 控制每秒抽多少帧做推理

所以“每隔几帧推一次”在当前系统里，等价于：

- 把 `sampleFps` 调低

### 14.2 为什么前端会感觉“100ms 一下子出来好几条”

因为：

1. `modelLatencyMs` 表示单帧推理时间
2. 它不等于相邻回调之间的固定时间间隔
3. 流缓冲、解码和异步回调会让结果成簇到达

因此看到：

- 一帧大概 `100ms`
- 但短时间连续收到多条结果

这是流式异步链路的正常现象。

### 14.3 当前联调建议

如果前端觉得：

- 结果太密
- 页面太乱
- GPU 压力偏大

建议优先改：

- `sampleFps`

推荐口径：

1. 稳定联调优先：
   - `sampleFps = 1`
2. 展示流畅与压力折中：
   - `sampleFps = 2`
3. 细粒度观察优先：
   - `sampleFps = 3`

不建议在当前阶段继续无节制提高：

- 推理频率

因为当前主要瓶颈不是“推理不够频繁”，而是“结果质量还不够稳定”。

## 20. 阶段结论：取消继续精炼，转向开源借鉴 + 迁移学习

### 20.1 当前决定

截至本轮精炼训练停止时，决定：

- **取消继续只依赖当前数据做精炼训练**

原因不是这条线完全无效，而是：

1. 它已经证明当前修复后框架能学出正 `TP`
2. 但后续提升开始波动
3. 在没有额外数据的前提下，继续纯精炼的边际收益开始下降
4. 当前更值得把精力转到：
   - 借鉴更成熟的开源项目思路
   - 利用迁移学习提升 backbone / fusion 的上限

### 20.2 为什么现在该转方向

当前事实是：

1. 框架硬伤已经修掉
2. 当前模型已经不再是“完全不会检”
3. 当前主要问题变成：
   - 暗光场景不稳
   - 空帧偏多
   - 误检压不下去

这些问题只靠：

- 继续在同一套轻量 backbone 上精炼

很可能会越来越难继续明显改善。

所以当前更合理的路线是：

- 用开源成熟思路提升“特征表达 + 融合质量 + 弱模态鲁棒性”

而不是继续仅靠调阈值和小修小补。

2. 优先把 backbone 升级到可利用预训练的版本
3. 在当前工程内继续训练与联调

- 官方训练口径会先去白边，再调整到训练尺度
- 因此后续不要只关注模型，还要复查当前本地数据准备链路是否和官方口径一致

2. `VisDrone2019`

官方仓库：

- `https://github.com/VisDrone/VisDrone-Dataset`

为什么值得做辅助预训练：

- 它虽然不是 RGB-T 数据集，但它是很强的无人机视角目标检测基准
- 官方仓库给出的规模包括：
  - `288` 个视频片段
  - `261,908` 帧
  - `10,209` 张静态图
  - 超过 `2.6M` 个框

最适合当前项目的用途：

1. 给 RGB 分支做无人机视角预训练
2. 给检测头补 aerial small-object 定位先验
3. 提前学会无人机视角下车辆 / 行人 / 背景的几何分布

使用边界：

- 它不是 thermal 数据集
- 不建议和主任务直接混成同一阶段训练
- 更适合作为 RGB backbone 或 detector head 的前置 warmup

3. `LLVIP`

官方主页：

- `https://bupt-ai-cz.github.io/LLVIP/`

为什么值得补进路线里：

- 它是成对可见光 / 红外低照数据集
- 官方主页说明包含 `30976` 张图像，即 `15488` 对严格时空对齐的样本
- 大多数样本处于很暗场景

它最适合当前项目的用途不是直接做五类车检，而是：

1. 训练低照条件下的模态质量判断
2. 做 RGB 低照增强 / thermal 对齐相关的辅助预训练
3. 验证低照鲁棒性设计是否真的有效

使用边界：

- 它主要标注的是行人
- 与当前五类车辆任务不一致
- 更适合做低照辅助任务或前置 warmup，不适合直接当最终主训练集

4. `M3FD`

官方仓库：

- `https://github.com/JinyuanLiu-CV/TarDAL`

为什么值得做辅助多光谱预训练：

- 官方仓库给出的 `M3FD` 为严格配准的 visible + infrared 数据
- 规模为 `4200` 对图像，标注 `6` 类目标：
  - `People`
  - `Car`
  - `Bus`
  - `Motorcycle`
  - `Lamp`
  - `Truck`

最适合当前项目的用途：

1. 给 thermal branch 和 fusion 模块做前置 warmup
2. 让模型先学会多模态检测而不是只盯当前单一数据分布
3. 在进入 `DroneVehicle-like` 精调前，先建立更稳的跨模态表征

使用边界：

- 不是无人机视角
- 类别定义与当前任务不完全一致
- 只能作为辅助预训练集，不能替代最终主任务评估

5. `FLIR` Thermal ADAS Dataset

官方页面：

- `https://www.flir.com/oem/adas/dataset/`

为什么值得作为 thermal / low-light 辅助数据：

- 官方页面说明其提供 fully annotated thermal + visible frames
- 当前公开数据规格包括：
  - `26,442` 个标注帧
  - `520,000` 个 bounding boxes
  - `15` 个类别
  - thermal 与 visible 一一对应

最适合当前项目的用途：

1. 给 thermal branch 做 objectness 预训练
2. 给暗光、雾、眩光等复杂条件做鲁棒性补充
3. 帮助模型学到更稳的“热目标是否存在”判断

使用边界：

- 这是车载道路场景，不是无人机场景
- 类别体系与当前项目不同
- 更适合做 thermal / objectness warmup，而不是直接迁入最终评估口径

#### C. 数据使用原则

如果后续真的要借开源数据，不建议直接把所有数据混在一起一把梭训练。

更稳的顺序是：

1. 先保持 `DroneVehicle-like` 为最终唯一评估口径
2. 先做 RGB 或 multispectral 的前置 warmup
3. 再回到当前主数据集做精调
4. 所有辅助数据都只服务于初始化、鲁棒性增强和表征学习

必须避免的做法：

1. 不做类别重映射就直接混训
2. 不做场景区分就把车载与无人机数据混成一个训练阶段
3. 用辅助数据集上的提升，直接当成当前主任务真正提升

#### D. 如果当前分数仍不够高，下一阶段建议顺序

建议按下面顺序推进：

1. 先把当前 `experiment_refine_from_best.yaml` 跑完并留档
2. 复核 `DroneVehicle` 官方数据口径与本地 `dronevehicle_like_refined` 的一致性
3. 优先做 `ResNet18` 或更强 backbone 的迁移学习
4. 给 RGB 分支增加 `VisDrone` 无人机视角预训练
5. 给 fusion / thermal 分支增加 `M3FD` 或 `FLIR` 的辅助 warmup
6. 如果低照仍是主要痛点，再引入 `LLVIP` 做低照专项辅助训练
7. 然后再借 `UA-CMDet` 的 uncertainty-aware / illumination-aware 思路
8. 如果误检仍高，再借 `CALNet` 的冲突纠正 fusion

如果只保留一句最务实的话：

- 当前分数如果继续提不动，优先补“更强 backbone + 更对口的预训练 / 辅助数据 + 更成熟的跨模态融合思路”，而不是继续长时间硬炼同一版轻量结构。
