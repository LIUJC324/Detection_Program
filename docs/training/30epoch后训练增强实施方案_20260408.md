# 30 Epoch 后训练增强实施方案（2026-04-08）

本文档用于固化当前 `rgbt_uav_detection` 在清洗数据集后的后续增强路线。

适用前提：

- 当前正在跑的新训练基线：
  - `outputs/train_runs/run_20260408_180334`
- 当前训练配置：
  - `configs/default.yaml`
- 当前数据根目录：
  - `datasets/dronevehicle_like_refined`
- 当前起点权重：
  - `weights/best.pt`

当前原则：

1. 先让这轮 `30 epoch` 跑完，拿到清洗数据后的第一版完整验证结果。
2. 后续增强按阶段逐步实施，每一轮只改一类主因素。
3. 每轮都必须保留可回溯的 run 目录、配置和 best/last 权重。

## 1. 当前基线的真实含义

这轮训练相较旧基线，已经完成了两类关键数据修复：

1. 去掉了原始图四周固定白边。
2. 不再直接用单一模态标注监督双模态输入，而是使用 `merged_union` 方式生成共享框。

这意味着：

- 这轮 `30 epoch` 的结果是“清洗数据后的第一版基线”
- 后续所有增强都应该以这轮结果为新的对照组

在这轮跑完前，不建议继续同时改：

- 模型结构
- 训练时长
- 增强策略
- 输入尺寸

否则很难判断提升到底来自哪一项。

## 3. 后续增强的实施顺序

下面的步骤按顺序做，不建议跳着改。

### 第一步：延长训练预算并补训练调度缺口

目标：

- 解决当前 `30 epoch` 偏短的问题
- 解决“从 checkpoint 恢复但 scheduler/warmup 不够合理”的问题

建议改动：

1. `train.epochs` 从 `30` 提到 `50` 或 `60`
2. 增加 warmup
3. 显式保存和恢复 scheduler 状态
4. 如果继续从已有权重恢复，保证学习率轨迹连续

预计修改文件：

- `configs/default.yaml`
- `scripts/train.py`
- `model/detector.py`（如需在 checkpoint 中补字段）

实施理由：

- 当前这轮虽然已经清洗了数据，但实际上只是在新数据口径上继续跑剩余 epoch
- 对新数据分布来说，训练预算仍然偏紧

建议先做这一轮，而不是先改模型结构。

### 第二步：提升有效 batch，并处理 BatchNorm 小 batch 问题

目标：

- 降低小 batch 导致的统计抖动
- 提高训练稳定性和框回归质量

建议改动：

1. 如果显存允许，`batch_size: 4 -> 8`
2. 如果显存不允许，保留 `batch_size=4`，在 `train.py` 加梯度累积，把有效 batch 提到 `8` 或 `16`
3. 视结果决定是否冻结 BN
4. 如果冻结 BN 效果一般，再考虑把关键 BN 改成 GroupNorm

预计修改文件：

- `configs/default.yaml`
- `scripts/train.py`
- `model/network/backbone.py`
- `model/network/fusion_module.py`

实施理由：

- 当前骨干和融合模块大量使用 `BatchNorm`
- 在小目标任务里，小 batch 往往会让定位分支更不稳定

### 第三步：减轻对小目标不友好的增强

目标：

- 减少随机裁剪对小目标上下文的破坏
- 让模型先学会稳定回归，再逐步增加激进增强

建议改动：

1. `random_crop_prob: 0.3 -> 0.1`
2. `crop_min_scale: 0.7 -> 0.85`
3. 保留 `horizontal_flip`
4. `color_jitter` 先不动

预计修改文件：

- `configs/default.yaml`
- `data/transforms.py`

实施理由：

- 当前目标本来就偏小
- 大幅随机裁剪更容易切掉车辆上下文，导致分类可能有感知，但回归不稳

### 第四步：保持长宽比，替换掉直接拉伸到 640x640

目标：

- 减少形变
- 让裁边后的有效区域尽量保留真实几何比例

建议改动：

1. 当前 `Resize` 是直接缩放到正方形
2. 下一步改成保长宽比的 `letterbox/pad`
3. 同时把框坐标在 pad 前后映射清楚

预计修改文件：

- `data/preprocess.py`
- `data/transforms.py`
- `service/core/predictor.py`

实施理由：

- 当前裁边后图像接近 `640x512`
- 直接拉到 `640x640` 会产生非均匀形变
- 对俯视小车、长车、目标间距判断都不友好

注意：

- 这一步会影响训练和推理两端
- 做之前先保留上一轮最佳权重做对照

### 第五步：提高输入尺寸

目标：

- 给小目标更多像素
- 提高小目标召回

建议改动顺序：

1. `input_size: 640 -> 800`
2. 如果显存和速度还能接受，再试 `960`

同时联动：

1. 相应降低单卡 batch
2. 用梯度累积保住有效 batch

预计修改文件：

- `configs/default.yaml`

实施理由：

- 小目标检测里，输入分辨率往往比轻微结构改动更直接
- 在当前白边已去掉的前提下，提高输入尺寸的收益会更干净

### 第六步：适度增加模型容量

目标：

- 在已经确认数据和训练流程基本合理后，再抬模型上限

建议改动：

1. `backbone_channels: [32,64,96] -> [48,96,128]`
2. `fpn_out_channels: 128 -> 160` 或 `192`

预计修改文件：

- `configs/default.yaml`
- `model/network/backbone.py`
- `model/network/neck.py`
- `model/network/head.py`

实施理由：

- 当前结构明显偏轻量化，优先照顾实时性
- 如果目标是把精度先做上去，后续要给网络更高表达能力

注意：

- 这一步不应早于前面几步
- 否则容易把“数据/训练没调通”的问题误判成“模型太小”

### 第七步：继续增强数据而不是盲目堆 epoch

目标：

- 在当前清洗版数据集跑通后，继续扩大有效监督

优先级建议：

1. 先把现有 `train.zip` 解压并接入
2. 统一同样的裁边与标注融合口径
3. 再考虑引入其他 RGBT 数据集

预计修改文件：

- `scripts/prepare_dronevehicle_like.py`
- `configs/default.yaml`
- 必要时新增数据转换脚本

实施理由：

- 当前样本量仍然偏小
- 但必须保证新数据经过同样的清洗和标注口径统一

## 4. 每一轮增强都要怎么做验证

每实施完一步，固定做下面几件事：

1. 完整训练一轮新 run
2. 保留 `config_resolved.yaml`
3. 保存 `best.pt` 和 `last.pt`
4. 运行一次 `scripts/evaluate.py`
5. 对比以下指标：
   - `map50`
   - `recall50`
   - `small_recall50`
   - 预测框是否仍大量退化为极小框或点框

如果某一步没有带来明显收益：

- 不要继续在这条支线上叠加更多改动
- 回退到上一轮最佳配置，再试下一种增强

---

## 5. 推荐的具体执行顺序

按推荐优先级排序：

1. 跑完当前 `30 epoch`
2. 延长训练预算 + 修 scheduler/warmup
3. 提高有效 batch / 处理 BN
4. 降低激进裁剪
5. 保持长宽比
6. 提高输入尺寸
7. 增加模型容量
8. 扩充数据集

这个顺序的核心原因是：

- 先修训练流程
- 再修输入几何
- 再抬模型上限
- 最后扩数据规模

## 6. 当前不建议立刻做的事

当前阶段先不建议立刻做：

1. 直接换大模型再重训
2. 直接改成旋转框检测
3. 同时引入多个新数据集
4. 一次性同时改输入尺寸、增强、模型宽度和训练时长

原因：

- 这样会让问题定位彻底失去可解释性
- 你没法知道到底是哪一步带来了提升或退化

## 7. 模型端在线标注视频流实施路线

这一节用于说明：

- 当前项目最终是否可以做“模型端直接回传带框视频流”
- 这件事和当前“后端回调结果 + 前端叠加渲染”有什么区别
- 后续训练和优化完成后，应该怎样逐步实现

### 7.1 当前默认链路仍是什么

当前默认链路仍然是：

1. 前端播放器负责播放视频或实时流
2. 模型端负责拉流/抽帧/推理
3. 模型端把检测结果回调给后端
4. 前端根据结果做框叠加

这条链路的优点是：

- 结构简单
- 视频播放和检测渲染职责清晰
- 不需要模型端再承担实时编码与推流压力

所以在训练效果还不稳定前，不建议立刻把主链路切成“模型端直接回传带框视频流”。

### 7.2 为什么后续仍然要规划在线标注

虽然当前默认方案是“前端叠加框”，但后续实现模型端在线标注仍然有价值，主要用于：

1. 演示场景下减少前端叠加逻辑复杂度
2. 避免不同前端缩放、坐标换算、画框风格不一致
3. 为后续“模型侧直接产出可视化视频”能力预留实现路线

这条路线的最终目标是：

`原始视频流 -> 模型端推理 -> 模型端叠加标注 -> 编码成新视频流 -> 前端播放器直接播放`

### 7.3 实现在线标注前必须满足的前提

这件事必须建立在下面几个前提之上：

1. 模型检测效果已经基本稳定  
   否则只是把不稳定框直接烧进视频里，无法提升最终体验。

2. 当前双路输入组织方式已经定型  
   包括：
   - 单路拼接输入
   - 或双路按时间戳自动配对输入

3. 坐标口径已经稳定  
   包括：
   - 原图像素坐标
   - 是否保留白边
   - 是否保持长宽比

4. 模型服务的推理吞吐已经明确  
   否则叠加与编码会进一步拉高延迟。

### 7.4 在线标注需要补的功能需求

后续若做模型端在线标注，至少需要补下面这些能力。

#### A. 带框帧渲染器

需要新增一层渲染模块，负责：

- 把检测框画到 RGB 帧上
- 画类别和置信度
- 统一颜色、字体和线宽
- 可选叠加时间戳、sessionId、FPS 等调试信息

建议新增位置：

- `service/rendering/overlay.py`

#### B. 带框视频编码与输出

模型端不能只输出原始检测结果，还要把渲染后的帧重新编码成视频流。

至少需要支持一种输出：

1. 本地 `mp4` 录制
2. 本地或远端 `HTTP-FLV`
3. 后续再看是否接入 `WebRTC`

建议优先级：

1. 先支持文件录制
2. 再支持持续推流

#### C. 会话级输出控制

在线标注不是所有会话都必须打开，因此要给会话层增加配置项，例如：

- `annotatedVideoEnabled`
- `annotatedVideoOutputMode`
- `annotatedVideoOutputUrl`
- `annotatedVideoCodec`
- `annotatedVideoBitrate`

建议修改位置：

- `service/core/schemas.py`
- `service/api/app.py`
- `configs/deploy.yaml`
- `configs/deploy_stable.yaml`

#### D. 结果和视频的同步策略

模型端回传带框视频后，还要明确：

- 是否仍保留结果回调
- 回调结果和视频帧之间怎么对齐
- 前端如果既拿视频又拿结构化结果，以谁为准

建议：

- 在线标注视频流是“可视化输出”
- 结构化回调仍然保留，用于：
  - 统计
  - 落库
  - 事件触发
  - 调试

### 7.5 坐标体系和归一化字段的建议

如果后续仍有“前端自己叠加”需求，建议同时输出两套坐标：

1. 原图像素坐标  
   用于后端留档、调试和离线分析。

2. 归一化坐标（`0~1`）  
   用于前端在不同播放器尺寸下稳定渲染。

建议新增字段而不是替换旧字段，例如：

- `bbox`: `[x1, y1, x2, y2]`
- `bbox_normalized`: `[x1n, y1n, x2n, y2n]`

这样做的好处是：

- 旧接口兼容
- 前端可以逐步切换
- 后端历史数据结构不必立刻迁移

注意：

- 如果最终前端只播放“模型端已烧框的视频”，归一化坐标的重要性会下降
- 但为了调试和兼容，仍建议保留

### 7.6 推荐的实施顺序

不建议一步到位直接做“模型端实时带框推流”。

推荐顺序如下：

1. 保持当前默认链路不变  
   即“结果回调 + 前端叠加”继续作为主方案。

2. 先补离线带框视频导出能力  
   用于演示兜底和可视化验证。

3. 再补模型端会话内的“按帧渲染 + 保存本地 mp4”  
   先验证渲染模块和输出模块是否稳定。

4. 再补“模型端带框实时推流”  
   把本地录制输出替换成持续编码和推流。

5. 最后再决定前端是否切换为直接播放模型端带框流  
   如果切换，结构化回调仍建议保留。

### 7.7 预计需要修改的文件范围

如果后续开始实现在线标注视频流，预计会涉及：

- `service/streaming/session_manager.py`
- `service/core/predictor.py`
- `service/core/schemas.py`
- `service/api/app.py`
- `configs/deploy.yaml`
- `configs/deploy_stable.yaml`
- 新增 `service/rendering/overlay.py`
- 必要时新增：
  - `service/streaming/annotated_stream_writer.py`
  - `scripts/export_or_record_annotated_stream.py`

### 7.8 需要提前明确的工程约束

实现前必须先和前后端明确下面这些问题：

1. 前端最终要的是：
   - 结构化结果叠加
   - 还是直接播放带框视频流

2. 如果是带框视频流：
   - 返回给前端的是 `HTTP-FLV`、`mp4` 还是其他协议

3. 延迟容忍度是多少：
   - `500ms`
   - `1s`
   - `2s+`

4. 视频画质优先还是延迟优先：
   - 高码率更清晰
   - 低码率更低延迟

5. 后端是否仍需要结构化结果回调：
   - 如果需要，视频流和回调都要保留

### 7.9 当前阶段的结论

当前阶段对在线标注视频流的建议是：

1. 先把训练和检测质量做稳
2. 继续以“回调结果 + 前端叠加”作为主链路
3. 把“模型端在线标注视频流”纳入下一阶段工程增强项
4. 等训练基线跑完并完成一轮参数增强后，再开始实现
