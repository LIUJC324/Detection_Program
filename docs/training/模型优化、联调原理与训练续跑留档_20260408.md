## 2. 前后端与模型端的真实闭环

当前系统正确理解方式不是“前端把视频给模型，模型直接把框画出来”，而是下面这条链路：

1. 前端或设备侧推送原始流。
2. 流媒体服务负责暴露 HTTP-FLV 地址。
3. 后端创建业务会话并生成 `sessionId`。
4. 后端调用模型端 `POST /v1/inference/stream/start` 或 `POST /v1/inference/video/start`。
5. 模型端异步创建会话，进入 `STARTING`。
6. 模型端在后台线程里做 `ffprobe`、`ffmpeg` 拉流、拆帧、检测。
7. 模型端把结果回调到后端。
8. 后端通过 WebSocket 或页面轮询把结果送给前端。
9. 前端播放器播放视频流，页面层叠加检测结果。

关键原则：

- 视频播放链路和检测链路不是一条链路。
- `start` 接口返回 `200` 只表示“异步会话已受理”，不表示“已经开始成功推理”。
- 前后端必须继续查询 `GET /v1/inference/session/{sessionId}`，直到状态进入 `RUNNING`。

## 3. 为什么 `14:20` 已经有 `timed out after 20 seconds`，后面却还能推理

这个现象不是“同一次会话超时后又成功”，而是“两次不同会话”。

日志证据在：

- `/home/liujuncheng/rgbt_uav_detection/outputs/service_logs/service.log`
- `/home/liujuncheng/rgbt_uav_detection/outputs/service_logs/error.log`

按时间线：

1. `2026-04-07 14:20:19`
   后端发起会话 `d416410a49ec4ec6a4c53c3754dee129`。
2. `2026-04-07 14:20:27`
   这个会话在 `STARTING` 阶段就被停止。
3. `2026-04-07 14:20:39`
   该会话后台线程里的 `ffprobe` 才真正超时，记录 `timed out after 20 seconds`。
4. `2026-04-07 14:23:47`
   后端发起另一条新会话 `d9421610214a479c92dde552bcc83a3e`。
5. `2026-04-07 14:23:58`
   这条新会话 `session_probe_ok`，随后进入 `RUNNING` 并持续处理多帧。

结论：

- `start` 的 `200` 不代表“已经拿到首帧”。
- `stop` 即使发生在 `STARTING` 阶段，后台正在执行的 `ffprobe` 仍然可能稍后以超时形式落日志。
- 后续能正常推理，指的是后面重试的新会话成功，不是同一次会话自恢复。

## 4. 流式启动失败的工程根因

当前流式链路的难点不在模型卷积本身，而在“首帧之前的媒体探测和解码启动”。

原始流程是：

1. `session_manager.py` 里先用 `ffprobe` 获取 `width/height`
2. 只有拿到尺寸后，才启动 `ffmpeg` 把流按 `rgb24 rawvideo` 持续送入推理线程
3. 再由 `_split_pair_frame()` 按 `pairLayout` 和 `rgbPosition` 拆成 `RGB` 与 `Thermal`

原先的稳定配置里，为了降低端到端延迟，使用了较激进的参数：

- 很短的 `ffprobe_timeout_seconds`
- `ffmpeg_low_latency = true`
- 偏小的 `analyzeduration`
- 偏小的 `probesize`

这会带来两个副作用：

- 流刚建立、关键帧还没到时，`ffprobe` 更容易超时。
- FLV 起始元数据不完整时，`ffmpeg` 更容易报：
  - `Packet mismatch`
  - `could not find codec parameters`

### 5.2 代码侧为 `ffprobe` 增加重试

文件：

- `/home/liujuncheng/rgbt_uav_detection/service/streaming/session_manager.py`

当前做法：

- `ffprobe` 对 HTTP 源也带 `rw_timeout`

- 流启动前的探测不再是一把失败就立刻判死

原则：

- 对实时流来说，“首帧未就绪”不等于“整条流不可用”
- 启动阶段需要容忍短时间不稳定

## 6. 模型当前问题的真实根因

### 6.1 当前权重还明显没训练够

训练指标证据在：

- `/home/liujuncheng/rgbt_uav_detection/outputs/train_runs/run_20260402_201142/latest_metrics.json`

当前最好一版历史记录：

- `epoch = 5`
- `map50 = 0.054545`
- `recall50 = 0.0`
- `small_recall50 = 0.0`

这说明当前权重的主要问题不是“已经学好了，只差后处理”，而是：

- 分类和回归都还没有收敛
- 特别是小目标回归还没有学稳定

### 6.2 为什么会出现很多非常接近的点框

当前模型用的是 `torchvision FCOS`，属于 Anchor-free 检测器。

在 FCOS 里，每个特征点都会输出：

- 类别分数
- 边框回归量
- centerness

如果分类支路已经能大致感知“这里附近有车”，但框回归支路还没学好，就会出现：

- 一堆相邻位置都给出相近分数
- 但它们的框回归塌缩到非常小，甚至退化成点

在这种情况下，普通 IoU NMS 效果很差，因为：

- 点框或极小框之间的 IoU 往往接近 `0`
- IoU NMS 无法把这些近邻重复响应有效压掉

所以你看到的现象是：

- 分数很多在 `0.5 ~ 0.7`
- 框中心非常密
- 但明显没那么多真实车辆

这不是前端画框的问题，而是当前权重本身的回归质量还不够。

## 7. 本轮对模型输出的临时止血方案

### 7.1 原则

不改 API 结构，不改前后端字段，仅在模型端内部加一个“退化点框近邻去重”的后处理。

### 7.2 代码位置

文件：

- `/home/liujuncheng/rgbt_uav_detection/service/core/predictor.py`

新增了三个部署配置项：

- `result_min_confidence`
- `result_merge_distance`
- `result_merge_degenerate_size`

当前稳定配置值：

- `result_min_confidence: 0.65`
- `result_merge_distance: 6`
- `result_merge_degenerate_size: 2`

### 7.3 算法原理

当前临时后处理的原则是：

1. 先按 `confidence` 过滤掉过低分结果。
2. 仅对“退化框”或“极小框”做近邻去重。
3. 只有同类别、中心点距离非常近的框才会被合并。
4. 对正常面积框不做这个去重，避免把相邻真实目标误删。

这里“退化框”指：

- `x2 - x1 <= result_merge_degenerate_size`
- 或 `y2 - y1 <= result_merge_degenerate_size`

这套规则的目标不是提升学术指标，而是临时降低：

- 一堆密集重叠点
- 前端肉眼看来明显不合理的多车响应

### 7.4 当前效果

在已知问题样例 `dronevehicle_00001` 上：

- 原始输出：`50` 个检测
- 临时后处理后：`8` 个检测

这说明止血有效，但它不代表“模型已经真正学好”。

## 8. 前后端联调需要同步的语义

### 8.1 `sourceUrl` 不是固定不变的

目前历史日志里模型端真实拉取过两类地址：

- 固定演示流：
  - `http://8.137.107.232:8088/live/001.flv`
- 动态拼接流：
  - `http://8.137.107.232:8088/bind/mix_bind_rgb_..._bind_ir_....flv`

联调时不能只盯住 `live/001.flv`，因为真正正在推理的往往是动态生成的 `bind/mix_...` 地址。

### 8.2 前端播放器看到画面，不代表模型一定已开始推理

原因：

- 播放链路只要求播放器能消费 HTTP-FLV
- 模型端还要先完成 `ffprobe -> ffmpeg -> 拆帧 -> 推理`

所以联调判断依据应该是：

1. 前端播放器是否有画面
2. `GET /v1/inference/session/{sessionId}` 是否进入 `RUNNING`
3. 后端是否收到模型回调
4. 前端是否收到后端推送的检测结果

### 8.3 `start` 返回 `200` 不是“成功开始推理”

这是后端必须统一的口径：

- `200` 只等于“模型端接受了异步会话创建请求”
- 真正的运行态必须通过 `GET session` 或回调事件确认

## 9. 继续训练的必要性

当前模型必须继续训练，理由如下：

1. 最佳权重实际上只训练到了 `epoch 5`。
2. 当前离线指标太低，远不到可交付状态。
3. 当前后处理只是压重复点，不能从根上修复框回归质量。
4. 如果不继续训练，前端看到的“分数偏低、点很密”问题会一直反复出现。

## 11. 当前最稳妥的结论

可以明确成立的结论有四条：

1. 流启动稳定性问题主要集中在 `ffprobe/ffmpeg` 首帧探测阶段，而不是服务主进程本身崩溃。
2. 当前模型的密集点框和中低分问题，主要根因是权重尚未训练到位。
3. 退化点框近邻去重可以作为临时止血方案，但不能替代继续训练。
4. 后端和前端联调时，必须以“会话状态”和“回调结果”判断模型是否真正开始工作，不能只看 `start 200` 或播放器是否有画面。
