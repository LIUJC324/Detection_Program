## 1. 项目需求确认

结合仓库根说明和当前代码，项目的真实需求不是“把一轮训练跑通”这么简单，而是下面这一套完整闭环：

### 1.1 任务目标

- 输入为无人机视角的 `RGB + Thermal` 双模态图像。
- 场景覆盖白天、夜间、低照、复杂天气等全天候交通场景。
- 检测类别固定为 `5` 类：
  - `car`
  - `truck`
  - `bus`
  - `van`
  - `freight_car`
- 不仅要支持离线训练与评估，还要支持导出、服务化部署和前后端联调。

### 1.2 工程目标

- 训练入口：`scripts/train.py`
- 评估入口：`scripts/evaluate.py`
- 导出入口：`scripts/export.py`
- 服务入口：`service/api/app.py`
- 目标不是只做单张图检测，而是最终接入实时视频/实时流检测闭环。

### 1.3 当前默认技术路线

- 检测器：`torchvision FCOS`
- 骨干：双流轻量 CNN
- 融合：跨模态注意力融合
- Neck：轻量 BiFPN 风格结构
- Head：小目标增强特征精炼头
- 训练配置：`configs/default.yaml`
- 当前训练数据：`datasets/dronevehicle_like_refined`

## 3. 当前指标不能直接当真值的原因

### 3.1 当前 `map50` 存在评估偏高风险

文件：

- `scripts/eval_utils.py`

关键问题：

- `_ap_from_ranked()` 在没有任何真阳性的情况下，只要某个类别有预测记录，仍可能返回非零 AP。
- 经过最小样例复核，`_ap_from_ranked([0, 0, 0], 10)` 当前会返回 `0.090909...`，这显然不合理。

这意味着：

- 当前看到的 `map50 = 0.072727` 不能直接当成“模型已经有稳定检测能力”的证据。
- 在 `recall50 = 0.0`、`small_recall50 = 0.0` 的前提下，优先要修评估，再决定模型优化方向。

### 3.2 当前更可信的结论

在现有评估实现下，能确定的只有：

- 模型对验证集 GT 还没有形成有效匹配
- 至少在当前阈值和当前实现下，`matched_gt = 0`
- 当前瓶颈不是“再微调一点分数就好”，而是“检测定位还没有真正学出来”

## 4. 数据与模型的结构性问题

### 4.1 类别严重不均衡

`dataset_summary.json` 记录的训练集分布：

- 总框数：`17098`
- `car = 14440`
- `truck = 1056`
- `bus = 587`
- `van = 505`
- `freight_car = 510`

这意味着：

- `car` 占比约 `84%`
- 其余四类明显偏少
- 如果不做类别重平衡，模型很容易向主类塌缩

### 4.2 当前数据并不是“绝大多数都极小目标”

今天对标注框面积做了统计：

- train：
  - `small <= 1024` 占比 `14.38%`
  - 中位面积 `1914`
- val：
  - `small <= 1024` 占比 `12.51%`
  - 中位面积 `1922`

这说明：

- 当前任务里确实有小目标，但不是“绝大多数都是极小目标”
- 因此单纯继续加大小目标损失权重，不一定能解决主问题
- 当前 `small_object_boost = 0.4` 更像辅助手段，不应被当成主优化方向

### 4.3 当前模型容量和融合策略偏轻

当前实现大致是：

- 双流轻量 backbone
- 每层一个较轻的 `CrossModalAttentionFusion`
- 再接轻量 BiFPN 和精炼 head

对比近期更强的 RGB-T / 多光谱检测器，当前差距主要在：

- 缺少更强的跨模态冲突建模
- 缺少不确定性建模
- 缺少更强的全局上下文交互
- 缺少更成熟的预训练骨干和训练框架

## 5. 可以直接借鉴的开源模型与框架

下面这些是今天筛过、与当前项目最相关的开源路线。

### 5.1 UA-CMDet

链接：

- https://github.com/SunYM2020/UA-CMDet
- https://github.com/VisDrone/DroneVehicle

为什么最值得优先学：

- 它就是基于 `DroneVehicle` 做的 RGB-Infrared 车辆检测
- 场景、类别和任务形式与当前项目最接近
- 明确引入了：
  - uncertainty-aware learning
  - illumination-aware cross-modal NMS

建议重点借鉴：

- 用目标级不确定性给训练样本加权
- 用光照/模态质量参与后处理和融合
- 学它的数据处理与 DroneVehicle 训练口径

### 5.2 CALNet-Dronevehicle

链接：

- https://github.com/hexiao0275/CALNet-Dronevehicle

为什么值得学：

- 直接针对“跨模态语义冲突”这个问题
- 提出了：
  - `CCR`：Cross-Modal Conflict Rectification
  - `SCF`：Selected Cross-modal Fusion

对当前项目最有价值的地方：

- 你现在的融合模块还比较轻，主要是显式 attention 加权
- CALNet 这类方法更强调“先消冲突，再做融合”
- 很适合替换或增强当前 `fusion_module.py`

### 5.3 CFT

链接：

- 论文页：`Cross-Modality Fusion Transformer for Multispectral Object Detection`
- 代码地址在公开资料中给出为：
  - https://github.com/DocF/multispectral-object-detection

为什么值得学：

- 它把全局上下文和跨模态交互提到了 Transformer 级别
- 比简单的局部卷积注意力更擅长建模长距离依赖

适合当前项目的借鉴点：

- 中高层特征融合方式
- 更强的跨模态交互，而不是只做点状加权

### 5.4 EAEFNet

链接：

- https://github.com/FreeformRobotics/EAEFNet

为什么值得学：

- 它不是只针对单一检测任务，而是一个更通用的 RGB-T 融合思路
- 重点强调“RGB 与 Thermal 有时都有效，有时只有一边有效”的显式融合逻辑

适合当前项目的借鉴点：

- 融合模块设计理念
- 异常模态、弱模态条件下的特征补偿

### 5.5 MMDetection

链接：

- https://github.com/open-mmlab/mmdetection
- https://mmdetection.com/

为什么必须列入路线：

- 当前自定义训练框架已经能跑通，但训练/评估/可视化/实验管理能力仍偏薄
- MMDetection 有成熟的：
  - model zoo
  - 预训练权重
  - loss / assigner / sampler
  - 评估与日志体系

建议把它当成：

- “更强 backbone 和 detector 的基座”
- 或者“后续把 RGB-T 融合模块移植进去的宿主框架”

## 6. 优化优先级与实施顺序

### 第一优先级：先修评估，再谈模型优劣

目标：

- 让 `map50 / recall50 / small_recall50` 变成可信指标

建议动作：

1. 修正 `scripts/eval_utils.py` 中 `_ap_from_ranked()` 的 AP 计算逻辑。
2. 修完后对 `weights/best.pt` 和 `weights/last.pt` 重新做离线评估。
3. 输出：
   - 总体 mAP50
   - per-class AP50
   - 真实 TP / FP / FN
   - 不同 score threshold 下的 recall 变化

不做这一步的风险：

- 后面所有“模型提高了还是没提高”的判断都可能是假的

### 第二优先级：先做诊断性实验，不要一上来大改结构

目标：

- 快速判断当前问题到底是阈值问题、类别问题、输入尺寸问题，还是融合问题

建议动作：

1. 验证时将推理 score threshold 从 `0.2` 下探到 `0.05`
2. 统计每类预测框数量、平均分数、TP 数量
3. 看当前模型是：
   - 完全没框
   - 有框但全错类
   - 有框但回归严重偏

说明：

- 如果阈值一降，recall 立刻起来，说明模型“有感知但输出太保守”
- 如果阈值降了仍然没有 TP，说明主要问题是定位/融合/监督本身

### 第三优先级：解决类别不均衡

目标：

- 避免模型被 `car` 类主导

建议动作：

1. 给分类损失加类别权重，或直接改成 class-balanced/focal 变体
2. 增加类别均衡采样
3. 离线评估必须输出 per-class AP，不能只看总分

最值得先观察的现象：

- 是否只有 `car` 有预测，其他类别几乎空白

### 第四优先级：调整几何与输入分辨率

目标：

- 提高车辆边界可辨识度和框回归质量

建议动作：

1. 不再直接强行拉伸到 `640 x 640`，改成保长宽比 `letterbox/pad`
2. 输入尺寸从 `640` 逐步试到 `800 / 896 / 1024`
3. 重新评估 `random_crop` 对 UAV 车辆检测是否过强

建议顺序：

1. 先改 letterbox
2. 再升输入尺寸
3. 最后再决定是否保留当前随机裁剪强度

### 第五优先级：提升融合模块和主干容量

目标：

- 解决当前轻量融合不足的问题

建议动作：

1. 第一阶段先借鉴 `CALNet` 或 `EAEF`，替换/增强当前融合模块
2. 第二阶段再考虑借鉴 `CFT` 的更强全局交互方式
3. 如果要走工程稳定路线，优先考虑迁移到 `MMDetection`，再做 RGB-T 双流改造

不建议的做法：

- 在现有评估有问题、类别不均衡未处理前，直接大改成复杂 Transformer 主干

### 第六优先级：补齐训练策略

目标：

- 提高收敛稳定性

建议动作：

1. 补 warmup
2. 显式保存和恢复 scheduler 状态
3. 训练总 epoch 提到 `50 ~ 60`
4. 如果显存不足，做梯度累积，提升有效 batch
5. 视 batch 大小决定是否改 BN 策略

## 7. 推荐的落地实验顺序

建议按下面顺序做，每次只改一类主因素：

### Phase A：把指标先做真

1. 修 `eval_utils.py`
2. 重评估 `best.pt` / `last.pt`
3. 出 per-class AP 与 PR 曲线

### Phase B：最小代价诊断

1. 下调验证阈值
2. 统计每类预测数量
3. 抽取若干 val 样本做可视化

### Phase C：最稳的第一轮增强

1. 保长宽比 resize
2. 输入尺寸提升到 `800` 或 `896`
3. 减轻随机裁剪
4. 类别重平衡

### Phase D：结构升级

1. 先借 `CALNet / EAEF`
2. 再决定是否引入 `CFT`
3. 如需大重构，迁移到 `MMDetection`

## 8. 当前最推荐的技术路线

如果目标是“尽快把分数做上去，同时保留工程可维护性”，今天给出的首选路线是：

1. 先修评估
2. 做诊断性评估，确认阈值/类别/定位三者谁是主瓶颈
3. 做类别重平衡 + 保长宽比 resize + 更大输入尺寸
4. 在此基础上，把当前融合模块升级为 `CALNet` 或 `EAEF` 风格
5. 后续如需更强基座，迁移到 `MMDetection` 上继续做

最不建议的路线是：

- 在当前指标还不可信的前提下，直接盲目追求“更大模型”或“更复杂 Transformer”
