# 第二次测试记录（2026-04-02）

## 1. 测试背景

第一次测试已经确认：

- 远程 `HTTP-FLV` 地址可读
- 模型端能主动拉流
- 模型端能完成至少 1 帧推理

第二次测试在这个基础上进一步验证：

1. 这条流是否确实是单路拼接视频
2. 拆分后的 RGB / Thermal 画面是否能导出给前端对照
3. 模型端能否持续处理多帧，而不是只成功一次

## 2. 测试输入

测试地址：

- `http://8.137.107.232:8088/live/001.flv`

测试参数：

- `pair_layout = side_by_side_h`
- `rgb_position = left`
- `sample_fps = 1.0`
- 模型运行设备：`CPU`

说明：

- 之所以切到 CPU，是为了不影响当时正在运行的 GPU 训练

## 3. 测试方法

本次测试分成两部分：

### 3.1 画面拆分验证

先从远程流中导出首帧，然后：

- 保存原始首帧
- 按左右拼接规则拆成 RGB / Thermal
- 对拆分出来的 RGB 图再跑一次检测
- 保存带检测框的预览图

这一步的目的，是让前端能够直接看到：

- 模型端实际拉到的画面
- 模型端怎样把它拆成双路
- 模型端检测时看到的具体图像

### 3.2 多帧连续处理验证

再启动一个流会话，目标是：

- 持续处理至少 `5` 帧
- 如果达到目标，就主动停止会话
- 观察中途是否报错

## 4. 测试结果

最终结果摘要如下：

```json
{
  "final_status": "STOPPED",
  "frames_processed": 6,
  "last_error": null,
  "image_size": [840, 712],
  "model_version": "best"
}
```

关键信息：

- 最终状态：`STOPPED`
- 实际处理帧数：`6`
- 错误信息：`null`
- 拆分后单路尺寸：`840 x 712`

这说明：

1. 模型端可以持续从该 `HTTP-FLV` 读取多帧
2. 当前拆分规则 `[RGB | Thermal]` 是可用的
3. 会话在达到目标后可以正常停止
