# 视频测试数据集建议

这份说明只回答一个实际问题：

`如果要尽快和前端做一个视频演示测试，应该用什么数据最合适？`

## 1. 先给结论

如果目标是：

- 尽快联调
- 尽快让前端有视频文件可播
- 尽快验证“播放器 + 检测框叠加”链路

最推荐的不是先去下载一个“完全匹配比赛任务”的公开视频数据集，而是：

1. 先用本地 `DroneVehicle` 图像序列合成测试 mp4
2. 同时保留公开视频数据集作为后续扩展验证

原因很简单：

- 公开“RGB-T + 无人机视角 + 交通场景 + 车辆小目标 + 直接可下载 raw video”的数据集很少
- 而你本地已经有最匹配当前类别体系的 `DroneVehicle`
- 前后端联调阶段更关心“视频能播、接口能通、框能叠加”，不一定一开始就需要严格科研级视频 benchmark

## 2. 最匹配当前项目语义的数据

### 2.1 DroneVehicle

最匹配你当前项目的原因：

- 无人机视角
- RGB + Infrared 双模态
- 车辆类别与当前项目最接近
- 本地已经有数据

但它的限制也很明确：

- 官方主数据形态是图像对，不是你现在最需要的原始视频文件

官方来源：

- DroneVehicle 官方仓库：https://github.com/VisDrone/DroneVehicle

## 3. 适合做“真实视频链路验证”的公开数据

### 3.1 NII-CU Multispectral Aerial Person Detection Dataset

优点：

- 官方明确提供 raw video files
- RGB + FIR 双模态
- 无人机视角
- 适合验证“模型端拉视频/拉流 -> 抽帧 -> 连续回调”这条工程链路

缺点：

- 目标主要是行人，不是交通车辆
- 场景更偏人员监测，不是交通路面

官方来源：

- NII-CU 官方站点：https://www.nii-cu-multispectral.org/

### 3.2 VisDrone-VID

优点：

- 官方提供无人机视频数据
- 场景里包含车辆
- 非常适合验证“无人机视频检测前后端联调”

缺点：

- 只有 RGB，没有 Thermal

官方来源：

- VisDrone 官方数据集仓库：https://github.com/VisDrone/VisDrone-Dataset

### 3.3 UAVDT

优点：

- 无人机交通视频
- 车辆场景更贴近交通任务
- 适合验证视频文件检测与前端展示

缺点：

- 只有 RGB，没有 Thermal

参考来源：

- UAVDT 数据说明：https://hyper.ai/en/datasets/17869

## 4. 适合做 RGB-T 视频参考、但任务不完全匹配的数据

### 4.1 KAIST Multispectral Pedestrian Benchmark

优点：

- RGB + Thermal
- 交通道路环境
- 官方资料完整

缺点：

- 车载视角，不是无人机视角
- 目标以行人为主

官方来源：

- KAIST 官方站点：https://soonminhwang.github.io/rgbt-ped-detection/

## 5. 适合小项目的推荐选择

如果按“投入最小、收益最大”排序，我建议：

### 方案 A：立刻联调

直接用本地 `DroneVehicle` 图像序列合成 mp4。

优点：

- 和你当前类别最一致
- 零额外下载成本
- 最适合先让前后端把播放和叠加跑通

缺点：

- 它是“伪视频”，更适合演示，不适合当真正视频 benchmark

### 方案 B：补真实双模态视频链路

下载 NII-CU raw videos 做工程验证。

优点：

- 真正的 RGB-T 视频
- 可以验证视频读取、抽帧、持续回调

缺点：

- 和车辆检测任务不一致

### 方案 C：补真实无人机交通视频链路

下载 VisDrone-VID 或 UAVDT 做视频展示验证。

优点：

- 真正无人机交通视频
- 更适合给老师或前端展示“这是无人机交通场景”

缺点：

- 没有 Thermal

## 6. 本地已经准备好的快速方案

我已经补了一个脚本：

- `scripts/make_dronevehicle_demo_videos.sh`

它会把本地 `DroneVehicle` 验证集图像序列合成三个 mp4：

- RGB 视频
- Thermal 视频
- RGB + Thermal 左右拼接视频

现在还会额外生成：

- RGB + Thermal 左右拼接的 `.flv` 文件

默认输出目录：

- `outputs/demo_video/`

执行方式：

```bash
cd /home/liujuncheng/rgbt_uav_detection
bash scripts/make_dronevehicle_demo_videos.sh
```

也可以调参数：

```bash
FPS=10 FRAME_COUNT=240 START_NUMBER=1 bash scripts/make_dronevehicle_demo_videos.sh
```

如果要验证“模型端主动通过 HTTP 拉取视频”，还可以继续执行：

```bash
cd /home/liujuncheng/rgbt_uav_detection
/home/liujuncheng/miniconda3/bin/python scripts/test_http_paired_video_session.py
```

这个脚本会：

- 在本地临时起一个 HTTP 服务
- 暴露 side-by-side `.flv`
- 让模型端自己通过 HTTP URL 去读取
- 验证拆帧和检测链路

## 7. 推荐你现在怎么做

最实用的顺序是：

1. 先生成本地 `DroneVehicle` demo 视频
2. 把这个视频给前端先做播放器和叠框联调
3. 如果后续还要做更像真的实时演示，再补 NII-CU 或 VisDrone/UAVDT

这样你不会被“公开数据集下载和格式适配”卡住当前进度。
