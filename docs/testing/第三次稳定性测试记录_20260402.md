# 第三次稳定性测试记录（2026-04-02）

## 1. 测试目标

前两次测试已经说明：

- 这条远程 `HTTP-FLV` 地址可被模型端读取
- 模型端可以把视频拆成 `RGB + Thermal` 图像对
- 短时处理多帧没有报错

第三次测试的目标更明确：

- 连续跑满 `5min`
- 观察模型端拉流是否出现明显卡死、中断、异常退出
- 统计整个测试过程中的处理帧数和停滞情况

## 2. 测试输入

测试地址：

- `http://8.137.107.232:8088/live/001.flv`

测试参数：

- 时长：`300s`
- `sample_fps = 1.0`
- `pair_layout = side_by_side_h`
- `rgb_position = left`
- 运行设备：`CPU`

说明：

- 使用 CPU 是为了不影响正在进行的 GPU 训练

## 3. 测试方法

使用脚本：

- `scripts/test_http_flv_stability.py`

脚本行为：

1. 启动模型端会话
2. 每秒轮询一次会话状态
3. 记录：
   - 当前状态
   - 已处理帧数
   - 是否已有最新结果
   - 是否出现错误
4. 满 `300s` 后主动停止会话
5. 输出时间线和最终 summary

## 5. 如何理解这些结果

### 5.1 正面结论

可以明确确认：

1. 整个 `5min` 过程中没有发生报错  
   `last_error = null`

2. 会话没有异常崩溃  
   最终是主动停止后的 `STOPPED`

3. 模型端可以持续拉取并处理这条 `HTTP-FLV`

4. 模型端始终能产出检测结果

### 5.2 需要注意的现象

虽然链路是通的，但它不是“完全满速无停顿”的。

从统计来看：

- `frames_processed = 223`
- 在 `300s` 的测试时间里，等效处理速度约 `0.74 fps`
- 存在一些轮询时刻帧数没有前进
- 最大连续停滞时间约 `11s`

这说明：

- 模型端并没有卡死
- 但这条链路在当前配置下仍然存在一定程度的处理间歇

## 6. 对“拉流卡不卡”的更准确结论

如果只问：

- 会不会拉不动？
- 会不会卡死？
- 会不会一会儿就断？

答案是：

- 当前 `5min` 测试看，`不会`

如果问更严格的：

- 是否完全平滑无抖动？
- 是否稳定达到 `1 fps` 采样目标？

答案是：

- `还没有`

更准确地说：

- 这条链路已经达到“可用”
- 但还没有达到“完全稳定、完全贴近目标采样率”

## 7. 这对前端意味着什么

前端同学说“播放器有点卡”，这次测试结果说明：

- 播放器卡，不等于模型端就拉不到
- 模型端这边已经能连续拉 `5min`
- 但模型端和播放器都表现出这条流存在一定抖动或处理间隙的可能

也就是说，现在更像是：

- 这条流整体上“能用，但体验还不算很稳”
