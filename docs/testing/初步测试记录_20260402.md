# 初步测试记录（2026-04-02）

## 1. 测试目标

验证下面这条链路是否已经打通：

- 模型端主动读取远程 `HTTP-FLV`
- 将单路视频帧按约定拆成 `RGB + Thermal` 图像对
- 复用当前检测模型完成推理
- 会话可正常启动、运行、停止

## 2. 测试输入

测试地址：

- `http://8.137.107.232:8088/live/001.flv`

测试假设：

- 视频为单路拼接视频
- 拼接方式为左右拼接
- 左半边是 RGB，右半边是 Thermal

对应参数：

- `pair_layout = side_by_side_h`
- `rgb_position = left`

## 3. 基础流探测结果

使用 `ffprobe` 探测得到：

- 容器格式：`flv`
- 编码：`h264`
- 分辨率：`1680 x 712`
- 平均帧率：`8 fps`

这个分辨率和本地 side-by-side demo 的尺寸一致，因此可以合理判断：

- 单帧大概率是 `[RGB | Thermal]`
- 拆分后每路尺寸约为 `840 x 712`

## 4. 模型端主动拉流测试方式

为了不影响正在进行的 GPU 训练，这次测试采用：

- 模型端使用 `CPU`
- `sample_fps = 0.2`
- 目标只验证“至少处理到 1 帧”

执行结果：

- 会话成功从 `STARTING` 进入 `RUNNING`
- 成功处理 `1` 帧
- 然后人工调用停止
- 最终状态为 `STOPPED`

## 6. 结论

本次初步测试可以确认：

1. 远程 `HTTP-FLV` 地址可被模型端读取
2. 当前代码已经能够主动拉取该流
3. 当前代码已经能够把该视频按“左右拼接”拆成 RGB-T 图片对
4. 检测推理链路已经能够正常跑通
5. 会话启动与停止链路正常

也就是说，至少在“模型端主动读取 `HTTP-FLV` 并完成初步推理”这一层，链路已经成立。

## 7. 当前边界

这次测试仍然只是“初步验证”，边界包括：

- 只验证了 1 帧处理成功
- 还没有做长时间稳定性测试
- 还没有验证断流重连
- 还没有验证后端 callback 联调
- 还没有验证前端播放器叠框闭环

## 8. 下一步建议

建议按下面顺序继续推进：

1. 让前后端按这条 `HTTP-FLV` 路径完成 start/stop 联调
2. 接入结果回调 `resultCallbackUrl`
3. 前端页面拿 WebSocket 或轮询结果叠框
4. 再做长时间稳定性测试
