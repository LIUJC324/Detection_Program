# 实时视频演示与前后端衔接说明

本文档不是讲模型结构本身，而是专门回答下面三个实际问题：

- 最终演示目标到底是什么
- 当前 ML 仓库已经实现到哪一步
- 你作为 ML 负责人，应该怎样和前后端同学对齐边界与接口

这份说明结合了：

- `require/【A18】面向全天候交通场景的无人机视角多模态小目标检测算法设计.pdf`
- `docs/integration/interface.md`
- 当前 `rgbt_uav_detection` 仓库里的已实现代码

## 1. 最终演示目标

最终目标不是只做一张图的检测，而是完成下面这条闭环：

1. 视频文件或实时流进入系统
2. 后端创建推理会话
3. ML 服务开始读取视频或拉流
4. ML 服务持续抽帧并做检测
5. 检测结果持续回调后端
6. 后端将结果推送给前端
7. 前端在播放画面上实时叠加检测框

如果一句话概括：

`最终要演示的是“实时视频检测系统”，而不是“离线图片检测脚本”。`

## 2. 当前仓库已经实现了什么

当前 `rgbt_uav_detection` 仓库已经实现的是：

- 数据读取、训练、评估、导出
- 单次 RGB-T 图像对推理
- 基于 FastAPI 的单次检测服务

目前真实可运行的服务接口只有：

- `GET /v1/health`
- `GET /v1/model/info`
- `POST /v1/detect/stream`

其中 `POST /v1/detect/stream` 实际含义是：

- 上传一对 RGB/T 图像
- 返回一次检测结果

它适合：

- 单帧测试
- 接口验收
- 模型调试

它还不等于最终演示所需的：

- 视频会话
- 实时流会话
- 连续推理
- 结果回调链路

## 3. 当前仓库还没实现但最终演示需要的部分

最终演示还至少需要下面这些能力：

### 3.1 会话控制

模型端需要能响应后端发来的：

- `startVideoSession`
- `startStreamSession`
- `stopSession`

### 3.2 视频或流读取

模型端需要能根据 `sourceUrl`：

- 读取视频文件
- 或拉取实时流
- 按固定频率抽帧

### 3.3 连续检测

模型端不能只做一次 `predict()`，而是要：

- 维护会话状态
- 持续读帧
- 持续输出检测结果

### 3.4 结果回调

模型端需要把检测结果回调给后端，而不是自己直接推前端。

### 3.5 双路 RGB-T 对齐

如果现场是双路 RGB/IR 流，必须解决：

- 两路帧时间对齐
- 两路分辨率对齐
- 异常流中断时的降级处理

这是最终系统里 ML 侧最容易被低估的工程点之一。

## 4. 最终系统里，ML 与前后端的职责边界

### 4.1 ML 负责什么

- 模型加载与推理
- 视频/流解码后的帧输入规整
- RGB-T 帧对齐
- 抽帧频率控制
- 检测结果生成
- 结果回调后端
- 模型健康检查、版本信息、推理耗时输出

### 4.2 后端负责什么

- 登录鉴权
- 设备管理
- 流路由管理
- 视频/流会话创建与停止
- 调用模型端 start/stop 接口
- 检测结果落库
- 检测结果转发 WebSocket
- 历史记录和统计报表

### 4.3 前端负责什么

- 页面交互
- 视频播放控件
- 会话切换
- 检测框渲染
- 实时结果列表
- 统计信息展示

## 5. 一定要和前端讲清楚的概念

前端最容易混淆的是：

- 播放视频
- 调用算法

这两件事不是同一条链路。

### 5.1 播放链路

前端播放器负责：

- 播放视频文件
- 或播放直播流

它的目标是“看到画面”。

### 5.2 检测链路

后端和模型端负责：

- 创建推理会话
- 抽帧检测
- 回传检测结果

前端拿到的是：

- 框坐标
- 类别
- 置信度
- 时间戳

然后把它叠加在播放器画面上。

所以可以直接对前端说：

`视频地址是给播放器的，检测结果是另一条实时消息链路给页面叠加的。`

## 6. 推荐的最终联调链路

建议采用下面这条链路，边界最清楚：

1. 前端发起“开始视频/开始流推理”
2. 后端创建 `sessionId`
3. 后端调用模型端启动接口
4. 模型端按 `sourceUrl` 读取视频或拉流
5. 模型端按固定 FPS 抽帧检测
6. 模型端把结果回调后端
7. 后端通过 WebSocket 推送前端
8. 前端播放器负责显示视频，页面层负责叠加检测框

这条链路的优点是：

- 模型端只专注推理
- 后端统一管理业务会话
- 前端只管展示

## 7. 推荐的小项目最小可交付方案

考虑到你们是小项目，建议不要追求“一上来就做全量直播平台”，而是优先做一个能稳定演示的最小版本。

推荐方案：

1. 保留现有 `POST /v1/detect/stream`  
   用于单帧冒烟和演示前调试。

2. 新增模型端会话式执行能力  
   最少支持：
   - 启动视频会话
   - 启动实时流会话
   - 停止会话

3. 抽帧而不是逐帧满帧推理  
   例如每秒 2 到 5 帧。

4. 结果异步回调后端  
   不要求模型端自己维持前端长连接。

5. 前端播放和检测叠加分离  
   播放器负责画面，WebSocket 负责框。

这是最适合小团队分工的形态。

## 8. ML 侧建议对齐给后端的接口口径

如果按 `docs/integration/interface.md` 的方案，ML 侧应准备好响应下面三类命令：

### 8.1 视频会话启动

后端传给 ML 的核心字段：

- `sessionId`
- `sourceUrl`
- `bucket`
- `objectKey`
- `resultCallbackUrl`
- `callbackToken`

### 8.2 实时流会话启动

后端传给 ML 的核心字段：

- `sessionId`
- `app`
- `streamKey`
- `sourceUrl`
- `resultCallbackUrl`
- `callbackToken`

### 8.3 停止命令

后端传给 ML 的核心字段：

- `sessionId`

## 9. ML 侧建议回调给后端的结果口径

建议至少包含：

- `session_id`
- `frame_index`
- `frame_timestamp`
- `detections`
- `inference_time`
- `image_size`
- `model_version`

如果后续要做时序分析，还可以加：

- `source_type`
- `stream_key`
- `is_end`

## 10. 你可以直接对前后端同学这样说

可以直接用下面这段口径：

`我这边 ML 先保证检测模型、抽帧推理、结果输出和回调接口稳定。最终演示里，视频播放不是由模型端负责，模型端只负责从 sourceUrl 取视频/流、按固定频率做检测、再把结果回调后端。前端播放视频走播放器链路，检测框走后端 WebSocket 推送链路。这样三边职责最清晰，也最适合小项目分工。`

## 11. 你当前最需要补的 ML 模块

如果从“当前仓库”走到“最终演示”，ML 侧最需要补的是：

1. 会话管理器  
   维护 `sessionId -> worker` 的映射关系。

2. 视频/流读取器  
   根据 `sourceUrl` 读取视频或拉流。

3. 抽帧调度器  
   控制推理频率，避免显存和算力被打满。

4. 结果回调器  
   将检测结果稳定回传给后端。

5. 异常处理  
   包括：
   - 源中断
   - 回调失败
   - 会话停止
   - 队列堆积

## 12. 结论

你现在的代码已经把“检测模型本体”和“单帧服务化接口”搭起来了。

下一步不是重写模型，而是补上“视频/流会话执行器”和“结果回调”这层工程壳。只要这层补齐，和前后端的系统链路就能闭环，最终就能支撑实时视频演示。
