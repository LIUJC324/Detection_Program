# 点标注过渡说明（2026-04-07）

说明：

- 本文档保留为阶段性变更记录
- 当前线上默认口径已切回 `rectangle`

## 1. 当前结论

为了先保证前后端联调和页面展示一致，模型端当前对“流/视频会话结果回调”采用：

- `point` 标注模式

具体含义：

- 模型内部仍然保留原始矩形框预测结果
- 但回调给后端的 `boxes` 暂时按“中心点”输出
- 因此当前回调里的：
  - `x1 == x2`
  - `y1 == y2`

这不是“点数据写坏了”，而是当前联调阶段的临时约定。

## 2. 哪些接口受影响

当前受影响的是：

- `POST /v1/inference/stream/start` 启动后的结果回调
- `POST /v1/inference/video/start` 启动后的结果回调
- 后端接收：
  - `POST /api/v1/detect/model/result`

当前不受影响的是：

- `POST /v1/detect/stream`

也就是说：

- 单帧检测接口里的 `detections[].bbox` 仍然是矩形框 `[x1, y1, x2, y2]`
- 会话回调里的 `boxes[].x1/y1/x2/y2` 当前是点标注语义

## 3. 当前回调语义

当前回调里新增：

- `annotationMode: "point"`

示例：

```json
{
  "taskId": "stream_demo_001:128",
  "sessionId": "stream_demo_001",
  "streamKey": "001",
  "annotationMode": "point",
  "boxes": [
    {
      "tag": "car",
      "score": 0.91,
      "x1": 144.7,
      "y1": 230.1,
      "x2": 144.7,
      "y2": 230.1
    }
  ],
  "modelLatencyMs": 42,
  "error": ""
}
```

## 4. 为什么先这样做

当前这样做的原因是：

1. 前后端先统一“点位叠加”的展示口径
2. 避免联调阶段继续争议矩形框缩放和渲染细节
3. 模型内部仍保留矩形预测，不会阻断后续升级

## 5. 后续升级为矩形标注怎么切

模型端已经保留升级入口。

当前配置项：

- `service.annotation_mode`

当前值：

- `point`

后续要切回矩形框时，只需要把配置改成：

- `rectangle`

对应配置文件：

- [deploy.yaml](/home/liujuncheng/rgbt_uav_detection/configs/deploy.yaml)
- [deploy_stable.yaml](/home/liujuncheng/rgbt_uav_detection/configs/deploy_stable.yaml)

切换后回调将恢复为：

- `x1, y1, x2, y2` 分别表示矩形框左上角和右下角

## 6. 前后端当前应如何处理

当前前后端联调阶段建议按下面理解：

1. 如果 `annotationMode=point`：
   - 把 `(x1, y1)` 视为目标点
   - `x2, y2` 只是为了兼容旧字段结构，和 `x1, y1` 相同
2. 如果未来 `annotationMode=rectangle`：
   - 按标准矩形框渲染

## 7. 兼容建议

前后端展示层建议直接按 `annotationMode` 分支：

1. `point`：
   - 画点
   - 或画一个固定像素大小的小十字/小圆点
2. `rectangle`：
   - 画标准矩形框

这样后续模型端升级为矩形标注时，前后端不需要再改接口字段名。
