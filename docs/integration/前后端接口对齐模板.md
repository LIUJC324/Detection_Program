# 前后端接口对齐模板


## 1. 先统一一句话口径


`ML 侧现在已经支持模型端主动拉取视频/流，不需要前端或后端逐帧把图片推给模型。只需要给模型端 source_url 和拼接方式，模型端会自己拉视频、拆成 RGB/Thermal 图像对并做检测。`

## 2. 发给前端同学的模板

下面这段可以直接转发给前端：

```text
现在 ML 侧已经支持“模型端主动拉视频/流”模式，不需要把帧一张张发给我。

需要和我对齐的字段有：
1. session_id：这次检测会话唯一标识
2. source_url：给模型端主动拉取的视频地址
3. pair_layout：视频里 RGB/IR 的拼接方式
   - side_by_side_h：左右拼接，[RGB | IR]
   - stacked_v：上下拼接，[RGB 在上，IR 在下]
4. rgb_position：
   - 左右拼接时：left / right
   - 上下拼接时：top / bottom
5. sample_fps：模型每秒抽几帧做检测，建议先 2~5

当前推荐你们给我的最小启动请求就是：
POST /v1/inference/stream/start

{
  "session_id": "demo_stream_001",
  "source_url": "http://host/live/abc123.flv",
  "sample_fps": 2,
  "pair_layout": "side_by_side_h",
  "rgb_position": "left"
}

说明：
- 视频播放链路和检测链路不是一回事
- 你们前端负责播放器播放
- 我这边负责主动拉 source_url、拆成 RGB/IR 图像对并检测
- 如果只是联调阶段，也可以先直接用本地 demo 视频 URL 测
```

## 3. 发给后端同学的模板

下面这段可以直接转发给后端：

```text
ML 侧现在已支持会话式主动拉流/读视频，后端不用转发逐帧图片，只需要给模型端下发启动命令。

建议对接字段：

1. 启动视频会话
POST /v1/inference/video/start

{
  "session_id": "video_demo_001",
  "source_url": "http://host/demo_side_by_side.mp4",
  "sample_fps": 2,
  "pair_layout": "side_by_side_h",
  "rgb_position": "left",
  "result_callback_url": "http://backend/api/v1/detect/model/result",
  "callback_token": "change-me"
}

2. 启动实时流会话
POST /v1/inference/stream/start

{
  "session_id": "stream_demo_001",
  "app": "live",
  "stream_key": "abc123",
  "source_url": "http://host/live/abc123.flv",
  "sample_fps": 2,
  "pair_layout": "side_by_side_h",
  "rgb_position": "left",
  "result_callback_url": "http://backend/api/v1/detect/model/result",
  "callback_token": "change-me"
}

3. 停止会话
POST /v1/inference/session/stop

{
  "session_id": "stream_demo_001"
}

4. 查询会话
GET /v1/inference/session/{session_id}

当前 ML 侧默认假设：
- source_url 对应的是单路拼接视频
- 每帧里已经拼好 RGB + IR
- 由 ML 侧按约定布局拆成图像对再做检测
```

## 5. 你可以现场解释给他们听的版本

如果是开会口头解释，可以直接用这版：

```text
RTMP/WebRTC/HTTP-FLV 说的是视频怎么进平台、怎么播、怎么被模型消费。
我们现在这边的约定是：
1. 设备把视频推到平台
2. 前端自己播放视频
3. 模型端自己主动拉 source_url
4. 模型端把每帧拆成 RGB/Thermal 图像对做检测
5. 检测结果再回到后端，最后叠加到前端页面

所以你们不用一帧一帧给我传图片，只要把会话参数和 source_url 对齐好就行。
```

## 6. 关键字段解释

### `session_id`

- 一次检测会话的唯一标识
- 便于后端查询状态、停止会话、聚合同一条流的检测结果

### `source_url`

- 模型端主动拉取视频/流的地址
- 当前代码已支持：
  - 本地或远程 mp4
  - HTTP 暴露的 flv
  - 其他 ffmpeg 可读取的视频源

### `pair_layout`

- `side_by_side_h`：左右拼接
- `stacked_v`：上下拼接

### `rgb_position`

- 左右拼接时：
  - `left`
  - `right`
- 上下拼接时：
  - `top`
  - `bottom`

### `sample_fps`

- 模型端每秒抽多少帧做检测
- 小项目联调时建议先用 `2` 到 `5`

## 7. 最小联调建议

如果你们还没把正式流媒体链路完全打通，建议先用最小方案：

1. 先用本地 side-by-side demo 视频
2. 前端先把播放器跑起来
3. 后端先把 start/stop/query 接口打通
4. 模型端先用 `source_url` 主动拉本地 HTTP 暴露的 `.flv`
5. 跑通后再替换成正式视频流地址
